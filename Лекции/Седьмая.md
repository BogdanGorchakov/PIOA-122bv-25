# Очередь, связанный список и рекурсия

## Введение

В этой лекции мы продолжим разговор о базовых структурах данных и алгоритмах. На прошлых занятиях мы уже обсуждали стек и подход TDD. Теперь логично перейти к структуре данных, которая во многом противоположна стеку, - очереди. После этого мы разберём связанный список как одну из удобных реализаций очереди, а затем перейдём к рекурсии и быстрой сортировке.

На первый взгляд это разные темы, но на самом деле они хорошо связаны между собой. Очередь и связанный список показывают, как выбор структуры данных влияет на эффективность операций. Рекурсия и быстрая сортировка показывают, как один и тот же алгоритм можно выразить через более "математический" способ мышления: базовый случай и рекурсивный переход.

## Очередь

### Общая характеристика

На прошлой лекции мы с вами рассматривали стек. Напомню, что стек - это структура данных, в которой доступ к элементам организован по принципу `LIFO` (`Last In, First Out` - последний пришёл, первый вышел). Например, стек можно увидеть в истории браузера: когда вы нажимаете кнопку "назад", вы возвращаетесь к последнему посещённому состоянию.

У стека есть структура данных, которая во многом является его противоположностью, - очередь. Очередь хранит элементы последовательно, но доступ к ним организован по принципу `FIFO` (`First In, First Out` - первый пришёл, первый вышел). Это поведение интуитивно понятно из реальной жизни: человек, который первым встал в очередь, первым и обслуживается.

Очереди встречаются очень часто:

- очередь задач в операционной системе
- очередь сообщений в сетевых приложениях
- очередь запросов к серверу
- очередь печати документов

То есть сама структура довольно простая, но при этом крайне практичная.

Это хороший инженерный подход: сначала формулируем контракт, потом заполняем его конкретным содержанием.

### Тестирование очереди

Давайте, как и в прошлой лекции, воспользуемся методологией TDD и сначала напишем тесты.

```python
# queue_test.py
from queue_example import Queue


def test_push():
    queue = Queue()
    queue.push(1)
    assert queue.size() == 1
    queue.push(2)
    assert queue.size() == 2


def test_pop():
    queue = Queue()
    queue.push(1)
    queue.push(2)
    assert queue.pop() == 1
    assert queue.size() == 1
    assert queue.pop() == 2
    assert queue.size() == 0

    queue = Queue()
    queue.push(1)
    queue.push(2)
    assert queue.peek() == 1
    assert queue.size() == 2
```

Такие тесты хороши тем, что сразу фиксируют поведение структуры данных. Мы не просто "думаем", как должна работать очередь, а формально записываем это в виде проверок. Дальше можно менять внутреннее устройство очереди, но если тесты продолжают проходить, значит внешний контракт не сломан.

## Очередь на массиве

### Реализация на списке Python

Сначала попробуем реализовать очередь с использованием встроенного списка Python.

```python
# queue_example.py
class Queue:
    def __init__(self):
        """
        Инициализация очереди. Используем список для хранения элементов очереди.
        """
        self.items = []

    def push(self, item):
        """
        Добавляет элемент в конец очереди.
        Сложность операции в лучшем случае O(1), в худшем случае O(n) из-за
        необходимости реаллокации списка при добавлении элемента.
        """
        self.items.append(item)

    def pop(self):
        """
        Удаляет и возвращает элемент из начала очереди.
        Сложность операции O(n) из-за необходимости сдвига всех элементов после
        удаления первого элемента.
        """
        return self.items.pop(0)

    def peek(self):
        """
        Возвращает элемент из начала очереди, не удаляя его.
        Сложность операции O(1), так как мы просто обращаемся к первому элементу
        списка.
        """
        return self.items[0]

    def size(self):
        """
        Возвращает количество элементов в очереди.
        Сложность операции O(1), так как мы просто возвращаем длину списка.
        """
        return len(self.items)
```

С точки зрения интерфейса всё выглядит хорошо: очередь работает корректно и удовлетворяет нашим тестам. Но здесь появляется важный вопрос эффективности.

У стека на списке Python операции добавления и удаления с конца обычно работают очень быстро. У очереди ситуация хуже, потому что удаление из начала массива требует сдвига всех остальных элементов. То есть структура ведёт себя правильно, но платит за это лишней работой.

Это важный урок: корректность и эффективность - не одно и то же. Можно написать правильную реализацию, которая при этом будет плохо масштабироваться на больших данных.

### Почему массив неидеален для очереди

Можно заметить, что все методы стека имели константную сложность, а у очереди на массиве метод `pop` уже работает за `O(n)`.

Причина в принципах работы динамического массива. Элементы лежат последовательно в памяти, и если удалить первый элемент, то все остальные приходится логически "сдвигать" влево. Именно поэтому массив отлично подходит для доступа по индексу, но не всегда удобен для операций в начале последовательности.

Чтобы избежать этой проблемы, можно использовать другие структуры данных. Первый вариант - кольцевой буфер.

### Кольцевой буфер

Кольцевой буфер представляет собой массив, в котором индексы как бы "зациклены". Когда мы доходим до конца массива, следующий индекс снова указывает на начало.

```text
Индексы:   0    1    2    3    4    5    6    7
Буфер:    [ ]  [ ]  [A]  [B]  [C]  [D]  [ ]  [ ]
                ^                   ^
               head                tail

Переход по индексам:
0 -> 1 -> 2 -> 3 -> 4 -> 5 -> 6 -> 7 -> 0 -> 1 -> ...
```

Идея кольцевого буфера в том, что мы больше не двигаем элементы физически. Вместо этого мы двигаем только указатели `head` и `tail`. За счёт этого операции вставки и удаления становятся `O(1)`.

```python
# queue_example.py
class Queue:
    def __init__(self, n=100):
        """
        Инициализация очереди. Используем кольцевой буфер для оптимизации
        операций push и pop.
        """
        self.items = [None] * n
        self.max_n = n
        self.head = 0
        self.tail = 0
        self._size = 0

    def push(self, item):
        """
        Добавляет элемент в конец очереди.
        Сложность: O(1), так как кольцевой буфер не требует сдвига элементов.
        """
        if self._size == self.max_n:
            raise IndexError("Queue is full")

        self.items[self.tail] = item
        self.tail = (self.tail + 1) % self.max_n
        self._size += 1

    def pop(self):
        """
        Удаляет и возвращает элемент из начала очереди.
        Сложность: O(1).
        """
        if self._size == 0:
            raise IndexError("Queue is empty")

        item = self.items[self.head]
        self.head = (self.head + 1) % self.max_n
        self._size -= 1
        return item

    def peek(self):
        """
        Возвращает элемент из начала очереди, не удаляя его.
        Сложность: O(1).
        """
        if self._size == 0:
            raise IndexError("Queue is empty")

        return self.items[self.head]

    def size(self):
        """
        Возвращает количество элементов в очереди.
        Сложность: O(1).
        """
        return self._size
```

Такое решение уже значительно лучше с точки зрения производительности. Но у него есть другой недостаток: размер буфера фиксирован заранее. Если мы не знаем, сколько элементов будет в очереди, то либо приходится заранее закладывать запас, либо думать о динамическом расширении буфера.

Это подводит нас к другой структуре данных - связанному списку.

## Связанный список

### Общая идея

Связанный список можно рассматривать как структуру, во многом противоположную массиву. Если использовать бытовую аналогию, то массив - это ряд ячеек, расположенных подряд, а связанный список - это набор ячеек, каждая из которых знает, где лежит следующая.

То есть элемент связанного списка хранит:

- само значение
- ссылку на следующий элемент

У последнего элемента ссылка на следующий узел отсутствует.

```text
[ A | * ] ---> [ B | * ] ---> [ C | * ] ---> [ D | None ]
```

Главное преимущество такого подхода в том, что вставка и удаление в начале структуры происходят очень дёшево: нам не нужно двигать все элементы, достаточно переназначить ссылки.

### Интерфейс связанного списка

Опишем интерфейс односвязного списка.

```python
# list_example.py
class ListExample:
    """
    Односвязный список.
    """

    def __init__(self):
        """
        Инициализация списка.
        Содержит ссылки на голову, хвост и размер списка.
        """
        self.head = None
        self.tail = None
        self._size = 0

    def append(self, value):
        """
        Добавление элемента в конец списка.
        """
        pass

    def pop(self, index):
        """
        Удаление элемента по индексу и возвращение его значения.
        """
        pass

    def __getitem__(self, index):
        """
        Получение элемента по индексу.
        """
        pass

    def __len__(self):
        """
        Получение количества элементов в списке.
        """
        pass


class Node:
    """
    Узел односвязного списка.
    """

    def __init__(self, value, next=None):
        """
        Инициализация узла. Содержит значение и ссылку на следующий узел.
        """
        self.value = value
        self.next = next
```

Здесь уже хорошо видно, как структура данных раскладывается на две сущности: сам список и отдельный узел. Список отвечает за общую организацию, а узел - за хранение одного элемента и ссылки на следующий.

Эти тесты важны ещё и потому, что связанный список устроен сложнее массива. Если в массиве многие операции "поддерживаются" уже самой структурой Python, то здесь мы полностью отвечаем за корректность ссылок. Ошибка в одном присваивании может повредить всю структуру.

### Реализация односвязного списка

Теперь напишем реализацию.

```python
# list_example.py
class ListExample:
    """
    Односвязный список.
    """

    def __init__(self):
        """
        Инициализация списка.
        Содержит ссылки на голову, хвост и размер списка.
        """
        self.head = None
        self.tail = None
        self._size = 0

    def _get_node(self, index):
        """
        Вспомогательный метод для получения узла по индексу.
        Сложность в лучшем случае O(1) для доступа к голове или хвосту,
        в худшем случае O(n) для доступа к элементу в середине списка.
        """
        if index < 0 or index >= self._size:
            raise IndexError("list index out of range")

        if index == 0:
            return self.head

        if index == self._size - 1:
            return self.tail

        current = self.head.next
        current_index = 1

        while current is not None and current_index < index:
            current = current.next
            current_index += 1

        return current

    def append(self, value):
        """
        Добавление элемента в конец списка.
        Сложность: O(1), так как мы используем ссылку на хвост.
        """
        new_node = Node(value)

        if self.head is None:
            self.head = new_node
            self.tail = new_node
        else:
            self.tail.next = new_node
            self.tail = new_node

        self._size += 1

    def pop(self, index):
        """
        Удаление элемента по индексу и возвращение его значения.
        Сложность: O(1) для удаления головы, O(n) для удаления из середины.
        """
        if index < 0 or index >= self._size:
            raise IndexError("list index out of range")

        if index == 0:
            removed = self.head
            self.head = self.head.next
            if self.head is None:
                self.tail = None
        else:
            previous = self._get_node(index - 1)
            removed = previous.next
            previous.next = removed.next
            if removed == self.tail:
                self.tail = previous

        self._size -= 1
        return removed.value

    def __getitem__(self, index):
        """
        Получение элемента по индексу.
        """
        return self._get_node(index).value

    def __len__(self):
        """
        Получение количества элементов в списке.
        """
        return self._size


class Node:
    """
    Узел односвязного списка.
    """

    def __init__(self, value, next=None):
        self.value = value
        self.next = next
```

### Сравнение массива и связанного списка

Теперь можно сравнить массивы и списки с точки зрения основных операций.

| Операция | Массив                                  | Связанный список                      |
| -------- | --------------------------------------- | ------------------------------------- |
| append   | O(1), если позволяет ёмкость, иначе O(n) | O(1)                                  |
| pop      | O(1), если удаляем последний элемент, иначе O(n) | O(1) для головы, иначе O(n) |
| []       | O(1)                                    | O(1) для головы и хвоста, иначе O(n)  |

Можно заметить, что связанный список показывает своё преимущество там, где нужно часто работать с началом структуры. При этом он заметно хуже для произвольного доступа по индексу. И это снова иллюстрирует главный принцип: универсально лучшей структуры данных не существует, выбор всегда зависит от набора операций.

## Очередь на связанном списке

### Реализация

Теперь вернёмся к очереди. Интерфейс у неё останется таким же, а вот внутренняя реализация изменится: вместо массива мы будем использовать односвязный список.

```python
from list_example import ListExample


class Queue:
    def __init__(self):
        """
        Инициализация очереди. Используем односвязный список.
        """
        self.items = ListExample()

    def push(self, item):
        """
        Добавляет элемент в конец очереди.
        Сложность операции O(1).
        """
        self.items.append(item)

    def pop(self):
        """
        Удаляет и возвращает элемент из начала очереди.
        Сложность операции O(1).
        """
        return self.items.pop(0)

    def peek(self):
        """
        Возвращает элемент из начала очереди, не удаляя его.
        Сложность операции O(1).
        """
        return self.items[0]

    def size(self):
        """
        Возвращает количество элементов в очереди.
        """
        return len(self.items)
```

### Почему это хорошая реализация

Можно заметить, что связанный список особенно хорошо подходит для очереди. Мы добавляем элементы в конец и удаляем элементы из начала - именно те операции, которые удобно поддерживаются через `head` и `tail`.

То есть здесь структура данных подобрана не "вообще хорошая", а хорошая именно под конкретный сценарий использования. Это очень важная привычка в программировании: выбирать реализацию не по красоте, а по характеру операций.

## Рекурсия

### Общая идея

Теперь перейдём к другой важной теме - рекурсии.

Многим из вас в целом понятна идея рекурсии: функция вызывает саму себя. Но важно понимать не только эту формальную часть, а сам способ мышления. Рекурсия особенно полезна в тех задачах, где проблему можно свести к уменьшенной копии самой себя.

Можно представить такую задачу: у вас есть коробка, а внутри неё могут лежать другие коробки. Нужно найти предмет. Здесь возможны два подхода:

- итеративный: складывать найденные коробки в некоторую коллекцию и постепенно их обрабатывать
- рекурсивный: если внутри коробки лежит другая коробка, просто повторить ту же самую процедуру для неё

Во втором случае логика становится очень естественной: "сделай то же самое, но для меньшей подзадачи".

### Базовый случай и рекурсивный случай

Давайте сначала посмотрим на простой пример рекурсии в Python.

```python
def rec(a=0):
    print(a)
    rec(a + 1)


rec()
```

Такая программа завершится с ошибкой `RecursionError: maximum recursion depth exceeded`. Причина проста: у рекурсии должен быть базовый случай, то есть ситуация, в которой функция перестаёт вызывать саму себя.

На предыдущих лекциях мы уже обсуждали стек вызовов. Каждый новый рекурсивный вызов добавляет новый кадр в стек. Python по умолчанию ограничивает глубину рекурсии, чтобы программа не переполнила стек вызовов. Текущее ограничение можно посмотреть через `sys.getrecursionlimit()`.

Чтобы программа работала корректно, нужно добавить условие выхода.

```python
def rec(a=0):
    # Базовый случай
    if a > 10:
        return

    # Рекурсивный случай
    print(a)
    rec(a + 1)


rec()
```

Здесь хорошо видно две обязательные части рекурсивного алгоритма:

- базовый случай - когда дальнейшие вызовы больше не нужны
- рекурсивный случай - когда задача сводится к более простой версии самой себя

### Факториал

Классический пример рекурсии - вычисление факториала.

Напомним, что:

- `n! = n * (n - 1)!`
- `0! = 1`
- `1! = 1`

То есть формула уже сама по себе подсказывает рекурсивную реализацию.

```python
def factorial(n):
    if n == 1 or n == 0:
        return 1

    return n * factorial(n - 1)


print(factorial(5))  # 120
```

Важно понимать, что рекурсия - это не "магия" и не отдельный тип вычислений. Любую рекурсию можно переписать через цикл.

```python
def factorial(n):
    if n == 1 or n == 0:
        return 1

    result = 1
    for i in range(2, n + 1):
        result *= i
    return result


print(factorial(5))  # 120
```

Поэтому рекурсия нужна не потому, что без неё задачу вообще нельзя решить, а потому, что в некоторых задачах она делает код короче, чище и ближе к математическому описанию.

## Быстрая сортировка

### Идея "разделяй и властвуй"

После рекурсии логично рассмотреть алгоритм, который особенно хорошо выражается рекурсивно, - быструю сортировку.

Быстрая сортировка основана на стратегии "разделяй и властвуй". Смысл стратегии в следующем:

- выделяется базовый случай, который решается совсем просто
- большая задача разбивается на более маленькие задачи того же типа
- решение большой задачи собирается из решений маленьких

Для тренировки можно рассмотреть даже не сортировку, а, например, сумму элементов массива.

```python
def sum_dc(items):
    if items == []:
        return 0

    return items[0] + sum_dc(items[1:])


print(sum_dc([1, 2, 3, 4, 5]))
```

Здесь базовый случай - пустой список, а рекурсивный шаг - сложение первого элемента с суммой остальных.

### Реализация быстрой сортировки

Теперь применим ту же идею к сортировке.

У быстрой сортировки:

- базовый случай - пустой массив или массив из одного элемента
- рекурсивный шаг - выбор опорного элемента и разбиение массива на части

```python
def quicksort(array):
    # Базовый случай
    if len(array) < 2:
        return array

    # Выбор опорного элемента
    pivot = array[0]

    # Элементы меньше опорного
    less = [item for item in array[1:] if item < pivot]

    # Элементы больше либо равны опорному
    greater = [item for item in array[1:] if item >= pivot]

    return quicksort(less) + [pivot] + quicksort(greater)


print(quicksort([10, 5, 2, 3]))
```

Обратите внимание на саму логику. Мы не пытаемся "в лоб" отсортировать весь массив сразу. Вместо этого мы выбираем опорный элемент и разбиваем массив на части, которые рекурсивно сортируются тем же самым методом.

### Сложность быстрой сортировки

В худшем случае сложность быстрой сортировки равна `O(n^2)`. Например, это может произойти, если массив уже отсортирован, а в качестве опорного элемента мы каждый раз выбираем первый элемент.

Для массива

`[1, 2, 3, 4, 5, 6, 7, 8]`

разбиения будут выглядеть примерно так:

1. `[] + [1] + [2, 3, 4, 5, 6, 7, 8]`
2. `[] + [2] + [3, 4, 5, 6, 7, 8]`
3. `[] + [3] + [4, 5, 6, 7, 8]`
4. `[] + [4] + [5, 6, 7, 8]`
5. `[] + [5] + [6, 7, 8]`
6. `[] + [6] + [7, 8]`
7. `[] + [7] + [8]`

Получается, что глубина рекурсии становится линейной, а на каждом уровне мы всё равно просматриваем почти весь оставшийся массив. Отсюда и возникает `O(n^2)`.

Но в среднем случае быстрая сортировка работает за `O(n * log n)`. Это происходит тогда, когда опорный элемент делит массив более-менее равномерно.

Например, если разбиение получается близким к центральному:

1. `[1, 2, 3] + [4] + [5, 6, 7, 8]`
2. `[1] + [2] + [3] + [4] + [5] + [6] + [7, 8]`

В этом случае глубина рекурсии растёт примерно как `log n`, а на каждом уровне мы всё ещё тратим `O(n)` на разбиение массива. Поэтому средняя сложность и получается `O(n * log n)`.
