# Решение третьего задания второй лабораторной работы

## Постановка задачи

Судя по тем решениям, которые я видел, нам с вами стоит отдельно обсудить третье задание второй лабораторной работы. Давайте сначала вспомним, что именно от нас требуется.

По условию нужно реализовать `in-memory` СУБД. Разберёмся, что это означает.

СУБД - это система управления базами данных. Если говорить о популярных СУБД, которые используются на практике, то можно вспомнить `PostgreSQL`, `MS SQL`, `Oracle`.

Типичная SQL-СУБД представляет собой набор таблиц. В случае нашей лабораторной работы у нас, например, есть таблица студентов.

| student_id | first_name | second_name | age | sex |
| ---------- | ---------- | ----------- | --- | --- |
| 1          | Иван       | Иванов      | 20  | М   |

Также у СУБД есть некоторый набор операций для работы с таблицами. Обычно этот набор называют `CRUD` от слов `create`, `read`, `update`, `delete`. Идея в том, что этих четырёх операций достаточно для базовой работы с данными.

Именно этот набор операций вам и нужно было реализовать в лабораторной работе. Создание записей и чтение уже были реализованы в методичке, и этого, по сути, хватало на базовый уровень. Обновление и удаление записей были задачей повышенной сложности, но в целом с ними многие справлялись нормально. Основные трудности возникли именно в третьем задании.

Если посмотреть на задание немного шире, то его смысл не только в том, чтобы "сделать ещё одну лабораторную". На самом деле это хорошее упражнение на моделирование данных. Вам нужно не просто написать несколько функций, а придумать, как в программе представить таблицу, запись, набор таблиц и правила работы с ними. Именно такие задачи и формируют инженерное мышление: сначала мы определяем структуру данных, потом описываем операции над ней, а затем уже думаем о расширении архитектуры.

В третьем задании требовалось добавить возможность создавать пользовательские таблицы. И здесь часто проявлялась одна и та же проблема: решение получалось слишком сложным. По-хорошему такие вещи действительно удобно реализовывать через классы, и к этому мы придём. Но сначала давайте посмотрим, как можно сделать это на обычных словарях.

### Почему начинаем со словарей

Если кратко, словарь - это коллекция, состоящая из пар ключ-значение. Подробнее со словарями мы ещё будем работать отдельно, а пока достаточно помнить две важные вещи:

- получение значения по ключу работает очень быстро, примерно так же удобно, как доступ по индексу в массиве
- в качестве ключа может выступать любой неизменяемый тип данных; в Python большинство примитивных типов неизменяемые, а вот списки и словари изменяемые

С точки зрения учебной логики словарь здесь удобен ещё и потому, что это самый простой способ собрать составную структуру данных без введения новых абстракций. Мы буквально "на руках" собираем базу данных из уже знакомых элементов языка: строки, кортежи, списки и словари. Такой подход полезен, потому что сначала позволяет увидеть модель данных в чистом виде, а уже потом переходить к более аккуратной архитектуре.

## Первая реализация на словарях

### Представление базы данных

Попробуем реализовать нашу базу данных на словарях.

Обратите внимание на саму идею структуры. У нас есть внешний словарь `Database`, который хранит набор таблиц по имени. Каждая таблица, в свою очередь, описывается двумя частями: схемой данных и самими строками. Это очень важный момент: даже в такой упрощённой реализации мы уже разделяем описание таблицы и содержимое таблицы.

```python
# Database - это словарь, в котором ключами являются имена таблиц, а значениями
# являются словари, описывающие структуру таблицы и её данные.
Database = {
    # Для примера используется одна таблица "Student", которая содержит
    # записи о студентах.
    "Student": {
        # Колонки таблицы представлены кортежем, который определяет
        # порядок и имена полей в каждой записи.
        "columns": ("student_id", "first_name", "second_name", "age", "sex"),
        "records": [
            # Каждая запись представлена словарём, в котором ключами
            # являются имена колонок, а значениями - соответствующие данные.
            {
                "student_id": 1,
                "first_name": "Иван",
                "second_name": "Иванов",
                "age": 20,
                "sex": "М"
            },
            {
                "student_id": 2,
                "first_name": "Мария",
                "second_name": "Петрова",
                "age": 22,
                "sex": "Ж"
            }
        ]
    }
}
```

### Альтернативный способ хранения строк

В этой реализации строки таблицы представлены словарями, где ключами выступают имена колонок. Вы могли выбрать и другой подход. Например, можно было бы хранить строки как кортежи, но тогда пришлось бы отдельно хранить порядок полей.

У каждого из этих подходов есть свои плюсы и минусы. Кортежи компактнее и ближе к "табличному" представлению, но хуже читаются и требуют помнить порядок колонок. Словари, наоборот, чуть более многословны, зато они нагляднее и проще для отладки. В учебной задаче это особенно важно, потому что нам важнее ясность структуры, чем микроскопическая экономия памяти.

```python
Database_with_tuples = {

            "student_id",
```

### Создание таблиц

Конкретный способ хранения данных остаётся на ваше усмотрение. Я дальше буду использовать первый вариант, где каждая запись хранится как словарь. Следующий шаг - добавить функцию создания новой таблицы.

Здесь важно заметить, что создание таблицы - это не просто добавление нового ключа в словарь. По сути, мы регистрируем новую сущность в нашей базе данных. У этой сущности есть имя, схема колонок и место для будущих записей. То есть уже на этом этапе база данных перестаёт быть просто "мешком данных" и начинает вести себя как система с определёнными правилами.

```python
def create_table(table_name: str, columns: tuple[str, ...]) -> None:
    """
    Создаёт новую таблицу с заданным именем и колонками.

    Если таблица с таким именем уже существует, возбуждается исключение ValueError.
    """

    # Проверка существования таблицы.
    if table_name in Database:
        raise ValueError(f"Таблица '{table_name}' уже существует.")

    # Создание новой таблицы.
    Database[table_name] = {
        "columns": columns,
        "records": []
    }
```

### Вставка и выборка записей

Теперь давайте адаптируем функции работы с данными под новую архитектуру таблиц.

Именно на этом этапе проявляется реальная сложность задачи. Пока структура хранится в виде обычных словарей, нам приходится вручную контролировать корректность почти на каждом шаге: существует ли таблица, все ли поля переданы, нет ли лишних полей, корректны ли фильтры. Это хороший пример того, как простая структура данных постепенно начинает требовать более строгой организации кода.

```python
from typing import Any


def insert_record(table_name: str, record: dict[str, Any]) -> None:
    """
    Вставляет новую запись в указанную таблицу.

    Выполняется проверка существования таблицы и соответствия полей записи структуре таблицы.
    В случае нарушения условий возбуждается исключение ValueError.
    """

    # Проверка существования таблицы.
    if table_name not in Database:
        raise ValueError(f"Таблица '{table_name}' не существует.")

    table = Database[table_name]

    # Проверка соответствия полей записи структуре таблицы.
    for column in table["columns"]:
        if column not in record:
            raise ValueError(f"Отсутствует поле '{column}' в записи.")

    # Проверка наличия дополнительных полей, которые не определены в структуре таблицы.
    for column in record:
        if column not in table["columns"]:
            raise ValueError(f"Поле '{column}' не определено в структуре таблицы.")

    # Вставка записи в таблицу.
    table["records"].append(record)


def select_records(table_name: str, **filters: Any) -> list[dict[str, Any]]:
    """
    Выполняет выборку записей из указанной таблицы в соответствии с переданными фильтрами.

    Если таблица не существует, возбуждается исключение ValueError.
    Если фильтры не заданы, возвращается копия всех записей таблицы.
    """

    # Проверка существования таблицы.
    if table_name not in Database:
        raise ValueError(f"Таблица '{table_name}' не существует.")

    table = Database[table_name]

    # Проверка отсутствия фильтров.
    if not filters:
        return table["records"].copy()

    # Формирование результирующего списка.
    result: list[dict[str, Any]] = []

    # Итерация по всем записям таблицы.
    for record in table["records"]:
        # Проверка соответствия каждому фильтру.
        # Если хотя бы один фильтр не подходит, переходим к следующей записи.
        match = True
        for key, value in filters.items():
            if key not in table["columns"]:
                raise ValueError(f"Поле '{key}' не определено в структуре таблицы.")
            if record.get(key) != value:
                match = False
                break

        # Если запись удовлетворяет всем заданным условиям,
        # она добавляется в результирующий список.
        if match:
            result.append(record)

    # Возврат списка найденных записей.
    return result


create_table("Course", ("course_id", "course_name", "instructor"))
insert_record("Course", {"course_id": 1, "course_name": "Математика", "instructor": "Петров"})
insert_record("Course", {"course_id": 2, "course_name": "Физика", "instructor": "Сидоров"})

results = select_records("Course", instructor="Петров")
print(results)  # [{'course_id': 1, 'course_name': 'Математика', 'instructor': 'Петров'}]
```

### Промежуточный вывод

Вот и всё. Никакой лишней функциональности здесь не нужно. Чем проще решение, тем лучше.

Но у такого подхода есть и недостатки. Нам приходится постоянно вручную проверять структуру данных, а сама структура таблицы никак не защищена от случайных изменений. Поэтому дальше мы естественным образом переходим к объектно-ориентированному подходу.

Именно в этот момент и становится видно, зачем вообще нужен ООП в прикладных задачах. Не ради "красоты" и не ради модного стиля, а ради того, чтобы перенести ответственность за корректность данных внутрь самих объектов. Когда программа растёт, это становится очень важным: чем меньше проверок разбросано по разным функциям, тем легче поддерживать код.

## Переход к объектно-ориентированному подходу

### Разделение ответственности между классами

Наш словарь довольно удобно раскладывается на два класса: саму базу данных можно оформить как класс `Database`, а отдельную таблицу - как класс `Table`. Давайте посмотрим, как это выглядит.

Такое разделение полезно тем, что каждый класс начинает отвечать за свою область. `Database` знает, какие таблицы вообще существуют и как к ним обращаться по имени. `Table` знает, как хранить записи и как выполнять операции над одной конкретной таблицей. Это и есть один из важнейших архитектурных принципов: разделение ответственности.

```python
from typing import Any


class Database:
    """
    Простая реализация базы данных в памяти.

    Класс позволяет создавать таблицы, вставлять записи
    и выполнять выборку по фильтрам.
    """

    def __init__(self):
        # Словарь таблиц базы данных.
        self.tables: dict[str, Table] = {}

    def create_table(self, table_name: str, columns: tuple[str, ...]) -> None:
        """
        Создаёт новую таблицу с заданным именем и колонками.

        Если таблица с таким именем уже существует, возбуждается исключение ValueError.
        """

        if table_name in self.tables:
            raise ValueError(f"Таблица '{table_name}' уже существует.")

        self.tables[table_name] = Table(columns)

    def insert_record(self, table_name: str, record: dict[str, Any]) -> None:
        """
        Вставляет новую запись в указанную таблицу.
        """

        if table_name not in self.tables:
            raise ValueError(f"Таблица '{table_name}' не существует.")

        self.tables[table_name].insert_record(record)

    def select_records(self, table_name: str, **filters: Any) -> list[dict[str, Any]]:
        """
        Выполняет выборку записей из указанной таблицы.
        """

        if table_name not in self.tables:
            raise ValueError(f"Таблица '{table_name}' не существует.")

        return self.tables[table_name].select_records(**filters)


class Table:
    """
    Класс Table описывает таблицу в базе данных.

    Он хранит информацию о колонках таблицы и её записях.
    """

    def __init__(self, columns: tuple[str, ...]):
        self.columns = columns
        self.records: list[dict[str, Any]] = []

    def insert_record(self, record: dict[str, Any]) -> None:
        """
        Вставляет новую запись в таблицу.

        Выполняется проверка соответствия полей записи структуре таблицы.
        В случае нарушения условий возбуждается исключение ValueError.
        """

        for column in self.columns:
            if column not in record:
                raise ValueError(f"Отсутствует поле '{column}' в записи.")

        for column in record:
            if column not in self.columns:
                raise ValueError(f"Поле '{column}' не определено в структуре таблицы.")

        self.records.append(record)

    def select_records(self, **filters: Any) -> list[dict[str, Any]]:
        """
        Выполняет выборку записей из таблицы в соответствии с переданными фильтрами.

        Если фильтры не заданы, возвращается копия всех записей таблицы.
        """

        if not filters:
            return self.records.copy()

        result: list[dict[str, Any]] = []

        for record in self.records:
            match = True
            for key, value in filters.items():
                if key not in self.columns:
                    raise ValueError(f"Поле '{key}' не определено в структуре таблицы.")
                if record.get(key) != value:
                    match = False
                    break

            if match:
                result.append(record)

        return result


db = Database()
db.create_table("users", ("id", "name", "age"))
db.insert_record("users", {"id": 1, "name": "Alice", "age": 30})
db.insert_record("users", {"id": 2, "name": "Bob", "age": 25})
print(db.select_records("users"))
print(db.select_records("users", name="Alice"))
```

### Зачем нам нужен следующий шаг

Теперь мы плавно подходим к четвёртой лабораторной работе. Там вам нужно будет реализовать работу с файлами. И здесь третья лабораторная естественным образом перетекает в четвёртую, потому что для поддержки разных способов хранения данных нам пригодится полиморфизм.

Другими словами, текущая лабораторная нужна не только сама по себе. Она подготавливает вас к следующему шагу. Если сейчас архитектура будет построена удачно, то позже вы сможете заменить хранение данных в памяти на хранение в файлах без полного переписывания программы. А это уже очень похоже на реальные задачи разработки, где систему постоянно приходится расширять, не ломая старый код.

## Подготовка архитектуры к полиморфизму

### Общий класс таблицы

Пусть у нас будет общий класс `Table`, который отвечает за поведение таблицы.

Здесь важно не запутаться в уровнях абстракции. Класс `Table` описывает не "конкретную таблицу users" или "конкретную таблицу products", а общий механизм работы любой таблицы: какие у неё есть колонки, как в неё вставлять записи и как из неё выбирать данные. Такой подход позволяет один раз описать общие правила и затем переиспользовать их.

```python
from typing import Any


class Table:
    """Таблица с фиксированным набором колонок и набором записей."""

    def __init__(self, columns: tuple[str, ...]) -> None:
        self.columns = columns
        self.records: list[dict[str, Any]] = []

    def insert_record(self, record: dict[str, Any]) -> None:
        """Вставляет запись, если её поля совпадают со схемой таблицы."""
        missing_columns = [column for column in self.columns if column not in record]
        if missing_columns:
            raise ValueError(f"Отсутствует поле '{missing_columns[0]}' в записи.")

        extra_columns = [column for column in record if column not in self.columns]
        if extra_columns:
            raise ValueError(
                f"Поле '{extra_columns[0]}' не определено в структуре таблицы."
            )

        self.records.append(record.copy())

    def select_records(self, **filters: Any) -> list[dict[str, Any]]:
        """Возвращает копии записей, удовлетворяющих фильтрам."""
        unknown_filters = [key for key in filters if key not in self.columns]
        if unknown_filters:
            raise ValueError(
                f"Поле '{unknown_filters[0]}' не определено в структуре таблицы."
            )

        if not filters:
            return [record.copy() for record in self.records]

        result: list[dict[str, Any]] = []
        for record in self.records:
            if all(record.get(key) == value for key, value in filters.items()):
                result.append(record.copy())

        return result
```

### Общий интерфейс базы данных

Теперь опишем интерфейс базы данных в целом. Идея в том, чтобы внешний код работал с единым набором методов, независимо от того, где именно хранятся данные.

Это очень важная мысль. Хорошая архитектура старается скрывать детали реализации от остальной программы. Пользовательскому интерфейсу, тестам или бизнес-логике обычно не должно быть важно, лежат данные в оперативной памяти, в файле или, например, во внешней СУБД. Если внешний контракт один и тот же, то внутреннюю реализацию можно менять значительно свободнее.

```python
from abc import ABC, abstractmethod
from typing import Any

from .table import Table


class Database(ABC):
    """Общий интерфейс базы данных с таблицами фиксированной структуры."""

    def create_table(self, table_name: str, columns: tuple[str, ...]) -> None:
        """Создаёт таблицу с заданным набором колонок."""
        if self._table_exists(table_name):
            raise ValueError(f"Таблица '{table_name}' уже существует.")

        self._save_table(table_name, Table(columns))

    def insert_record(self, table_name: str, record: dict[str, Any]) -> None:
        """Вставляет запись в таблицу."""
        table = self._load_table(table_name)
        table.insert_record(record)
        self._save_table(table_name, table)

    def select_records(self, table_name: str, **filters: Any) -> list[dict[str, Any]]:
        """Возвращает записи из таблицы, удовлетворяющие фильтрам."""
        table = self._load_table(table_name)
        return table.select_records(**filters)

    @abstractmethod
    def _table_exists(self, table_name: str) -> bool:
        """Проверяет, существует ли таблица."""

    @abstractmethod
    def _load_table(self, table_name: str) -> Table:
        """Загружает таблицу по имени."""

    @abstractmethod
    def _save_table(self, table_name: str, table: Table) -> None:
        """Сохраняет таблицу."""
```

### Реализация `MemoryDatabase`

Опишем конкретную реализацию `in-memory` базы данных.

По сути, это уже знакомая нам версия базы данных, только оформленная через более строгий интерфейс. Внутри всё ещё используется обычный словарь таблиц, но теперь этот словарь скрыт за методами класса. Это хороший пример того, как одна и та же идея может остаться прежней по сути, но стать гораздо удобнее по архитектуре.

```python
class MemoryDatabase(Database):
    """База данных, которая хранит таблицы в оперативной памяти."""

    def __init__(self) -> None:
        self.tables: dict[str, Table] = {}

    def _table_exists(self, table_name: str) -> bool:
        return table_name in self.tables

    def _load_table(self, table_name: str) -> Table:
        if table_name not in self.tables:
            raise ValueError(f"Таблица '{table_name}' не существует.")
        return self.tables[table_name]

    def _save_table(self, table_name: str, table: Table) -> None:
        self.tables[table_name] = table
```

### Демонстрационная реализация `FileDatabase`

А теперь сделаем простую демонстрационную заглушку для файловой базы данных.

Здесь пока нет полноценной работы с файлами, и это нормально. На данном этапе нам важен не сам механизм записи на диск, а форма взаимодействия с базой данных. Мы заранее проектируем архитектуру так, чтобы у нас была точка расширения. Позже внутрь этих методов можно будет добавить реальную файловую логику.

```python
class FileDatabase(Database):
    """Пример базы данных, которая сохраняет таблицы в файлах (демонстрация)."""

    def __init__(self) -> None:
        print("FileDatabase: демонстрация инициализации файловой базы данных.")

    def _table_exists(self, table_name: str) -> bool:
        print(f"FileDatabase: демонстрация проверки существования таблицы '{table_name}'.")
        return False  # Заглушка для демонстрации

    def _load_table(self, table_name: str) -> Table:
        print(f"FileDatabase: демонстрация загрузки таблицы '{table_name}'.")
        return Table(("id", "name", "price"))  # Заглушка для демонстрации

    def _save_table(self, table_name: str, table: Table) -> None:
        print(f"FileDatabase: демонстрация сохранения таблицы '{table_name}'.")
```

### Пример использования

И покажем, как с этим можно работать.

Обратите внимание на самое главное: код использования базы данных выглядит одинаково независимо от реализации. Мы вызываем `create_table`, `insert_record`, `select_records`, и именно это является признаком удачно выбранного интерфейса. Чем меньше клиентский код знает о внутреннем устройстве системы, тем легче систему менять и развивать.

```python
from .backend.database import FileDatabase, MemoryDatabase


if __name__ == "__main__":
    memory_db = MemoryDatabase()
    memory_db.create_table("users", ("id", "name", "age"))
    memory_db.insert_record("users", {"id": 1, "name": "Alice", "age": 30})
    memory_db.insert_record("users", {"id": 2, "name": "Bob", "age": 25})

    print("MemoryDatabase:")
    print(memory_db.select_records("users"))
    print(memory_db.select_records("users", age=30))

    file_db = FileDatabase()
    file_db.create_table("products", ("id", "name", "price"))
    file_db.insert_record("products", {"id": 1, "name": "Laptop", "price": 999.99})
    file_db.insert_record(
        "products", {"id": 2, "name": "Smartphone", "price": 499.99}
    )

    print("\nFileDatabase:")
    print(file_db.select_records("products"))
    print(file_db.select_records("products", price=499.99))
```

## Практический смысл такой архитектуры

### Что даёт единый интерфейс

Можно заметить, что в такой архитектуре использование баз данных становится логичнее. Мы больше не работаем с таблицами напрямую, а внутренняя реализация скрыта внутри класса базы данных. При этом интерфейс у разных баз данных одинаковый, а значит одну реализацию можно подменять другой без переписывания клиентского кода.

С практической точки зрения это означает, что мы начинаем проектировать программу "от контракта". Сначала определяем, какие действия должны быть доступны пользователю кода, а уже потом решаем, как именно они будут реализованы. Такой подход делает код устойчивее к изменениям и обычно сильно упрощает тестирование.

### Использование в пользовательском интерфейсе

Например, это удобно использовать в пользовательском интерфейсе. Можно дать пользователю выбор, с какой базой данных работать, но сам код интерфейса при этом почти не изменится.

Это хороший пример связи между архитектурой и удобством дальнейшей разработки. Если бы логика работы с памятью и файлами была жёстко зашита прямо в интерфейс, нам пришлось бы писать отдельные ветки кода почти для каждой операции. Но когда интерфейс базы данных единый, слой взаимодействия с пользователем остаётся простым и почти ничего не знает о деталях хранения данных.

```python
from .backend.database import FileDatabase, MemoryDatabase


class TUI:
    """Простой текстовый интерфейс для взаимодействия с базой данных."""

    def __init__(self) -> None:
        print("Выберите тип базы данных:")
        choice = input("1. MemoryDatabase\n2. FileDatabase: ").strip()
        if choice == "1":
            database_instance = MemoryDatabase()
        elif choice == "2":
            database_instance = FileDatabase()
        else:
            print("Неверный выбор. Используется MemoryDatabase по умолчанию.")
            database_instance = MemoryDatabase()
        self.database = database_instance

    def run(self) -> None:
        print("Текстовый интерфейс запущен. Введите команды для взаимодействия с базой данных.")

        while True:
            print("1 - создать таблицу, 2 - вставить запись, 3 - выбрать записи, q - выйти.")
            command = input("Введите команду: ").strip()
            if command == "1":
                args = input("Введите имя таблицы и колонки (через пробел): ").strip().split()
                self._create_table(args)

    def _create_table(self, args: list[str]) -> None:
        if len(args) < 2:
            print("Недостаточно аргументов для создания таблицы.")
            return

        table_name = args[0]
        columns = tuple(args[1:])
        self.database.create_table(table_name, columns)
        print(f"Таблица '{table_name}' с колонками {columns} создана.")
```

## Итог

Именно примерно такую логику я и ожидаю увидеть в ваших лабораторных работах. Здесь важна не сложность архитектуры сама по себе, а то, насколько она соответствует задаче. Если задачу можно решить просто и понятно, именно так и нужно делать.

Если кратко, то основная идея этой лекции такая: сначала мы строим рабочую модель на самых простых структурах данных, затем замечаем её ограничения, после чего постепенно приходим к более сильной архитектуре. Это правильный инженерный путь. Не нужно сразу начинать с избыточно сложных решений. Намного важнее уметь сделать простой рабочий вариант, увидеть его пределы и только после этого вводить новые абстракции там, где они действительно нужны.
