## Перегрузка операторов

### Общая характеристика

Вернемся к вопросам объектно-ориентированного программирования. В Python предусмотрена возможность перегрузки стандартных операторов, таких как сложение, умножение, сравнение и другие. Для этого в классах используются так называемые магические методы (`magic methods`), также называемые `dunder methods`.

Следующий пример демонстрирует реализацию класса двумерного вектора с перегрузкой нескольких операторов.

```python
class Vector:
    """Простой двумерный вектор.

    Класс хранит две координаты: `x` и `y`.
    Для объектов этого класса мы перегрузим несколько операторов,
    чтобы с ними было удобно работать почти так же, как с числами.
    """

    def __init__(self, x: float, y: float) -> None:
        """Создаёт новый вектор с координатами x и y."""
        self.x = x
        self.y = y

    def __add__(self, other: "Vector") -> "Vector":
        """Перегружает оператор `+`.

        Когда Python встречает выражение:
            vector1 + vector2
        он фактически вызывает:
            vector1.__add__(vector2)

        Здесь мы определяем правило сложения двух векторов:
        их соответствующие координаты складываются отдельно.
        """
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, number: float) -> "Vector":
        """Перегружает оператор `*`.

        В данном примере вектор умножается на число.
        Это означает, что каждая координата вектора
        умножается на переданный множитель.

        Пример:
            Vector(2, 3) * 4 -> Vector(8, 12)
        """
        return Vector(self.x * number, self.y * number)

    def __eq__(self, other: object) -> bool:
        """Перегружает оператор `==`.

        Метод отвечает за проверку равенства двух объектов.
        Сначала убеждаемся, что второй объект действительно является
        экземпляром класса Vector. Если нет, считаем объекты неравными.

        Затем сравниваем их координаты.
        """
        if not isinstance(other, Vector):
            return False

        return self.x == other.x and self.y == other.y

    def __getitem__(self, index: int) -> float:
        """Перегружает обращение к объекту по индексу.

        Метод `__getitem__` вызывается, когда мы пишем:
            vector[0]
        или:
            vector[1]

        Это позволяет обращаться к объекту так,
        будто он является списком или кортежем.

        В нашем примере:
        - `vector[0]` возвращает координату `x`
        - `vector[1]` возвращает координату `y`

        Если передан любой другой индекс,
        возникает ошибка IndexError.
        """
        if index == 0:
            return self.x
        if index == 1:
            return self.y
        raise IndexError("У вектора доступны только индексы 0 и 1")

    def __str__(self) -> str:
        """Возвращает строковое представление объекта.

        Этот метод используется, когда объект выводится через `print()`.
        Благодаря ему вместо стандартного технического вида объекта
        пользователь увидит понятную запись вектора.
        """
        return f"Vector({self.x}, {self.y})"



# Создаём два объекта класса Vector.
first_vector = Vector(2, 3)
second_vector = Vector(4, 1)

# Оператор `+` вызывает метод __add__.
# В результате получаем новый вектор,
# координаты которого равны сумме координат двух исходных векторов.
sum_vector = first_vector + second_vector

# Оператор `*` вызывает метод __mul__.
# Здесь мы умножаем первый вектор на число 3.
scaled_vector = first_vector * 3

# Оператор `==` вызывает метод __eq__.
# Проверяем, равен ли результат сложения вектору с ожидаемыми координатами.
is_equal = sum_vector == Vector(6, 4)

# Обращение по индексу вызывает метод __getitem__.
# Так можно получить отдельные координаты вектора.
first_coordinate = first_vector[0]
second_coordinate = first_vector[1]

# Выводим результаты работы программы.
# Благодаря методу __str__, объекты класса Vector будут отображаться в удобном виде.
print("Первый вектор:", first_vector)
print("Второй вектор:", second_vector)
print("Сумма векторов:", sum_vector)
print("Первый вектор, умноженный на 3:", scaled_vector)
print("Результат проверки sum_vector == Vector(6, 4):", is_equal)
print("Координата first_vector[0]:", first_coordinate)
print("Координата first_vector[1]:", second_coordinate)

```

### Механизм вызова магических методов

Когда интерпретатор Python встречает конструкцию `a + b`, он последовательно выполняет следующие действия:

- проверяет, определен ли у объекта `a` метод `__add__`
- если метод определен, вызывает его
- если метод не определен, пытается вызвать `b.__radd__(a)`

Методы с префиксом `r` представляют собой так называемые обратные версии соответствующих операций.

В рассматриваемом примере может быть добавлен также метод `__rmul__`, обеспечивающий корректную запись, в которой число расположено слева от объекта.

```python
def __rmul__(self, number: float) -> "Vector":
        """Перегружает оператор `*` для случая, когда число стоит слева.

        Этот метод вызывается, когда мы пишем:
            3 * vector
        Он позволяет нам умножать вектор на число, даже если число стоит перед 
        вектором.
        """
        return self.__mul__(number)
```

После добавления такого метода становится допустимой запись вида `scaled_vector = 3 * first_vector`.

### Часто используемые магические методы

На практике особенно часто используются следующие магические методы:

| Метод          | Использование  | Назначение                             |
| -------------- | -------------- | -------------------------------------- |
| `__init__`     | `obj = A()`    | Инициализация объекта                  |
| `__str__`      | `print(obj)`   | Человекочитаемое представление объекта |
| `__add__`      | `a + b`        | Сложение объектов                      |
| `__eq__`       | `a == b`       | Проверка равенства                     |
| `__len__`      | `len(obj)`     | Длина объекта                          |
| `__getitem__`  | `obj[i]`       | Получение элемента по индексу          |
| `__setitem__`  | `obj[i] = x`   | Установка элемента                     |
| `__contains__` | `x in obj`     | Проверка наличия элемента              |
| `__iter__`     | `for x in obj` | Итерация по объекту                    |

## Тестирование

### Назначение модульных тестов

Рассмотрим тестирование на примере кода из лабораторных работ. Предположим, что разработана простая СУБД с одной таблицей студентов, а также реализованы функции добавления и чтения записей. Даже если при ручной проверке программа работает корректно, это не гарантирует отсутствие ошибок после дальнейших доработок.

Типичная ситуация заключается в следующем: после добавления нового функционала ранее работавшие возможности перестают функционировать корректно. Такие ошибки называются регрессионными. Для снижения вероятности их появления применяются модульные тесты (`unit tests`).

Благодаря декомпозиции программы на небольшие и изолированные подпрограммы появляется возможность тестировать отдельные функции независимо. В общем случае тест представляет собой отдельную функцию, которая вызывает тестируемую подпрограмму с определенным набором параметров или состояний и проверяет, что результат соответствует ожидаемому.

```python
# Например функция тестирования создания записей
def test_create_record():
    # Очистка таблицы перед тестированием.
    Student.clear()

    # Тест 1: Создание валидной записи.
    record1 = create_record(1, "Иван", "Иванов", 20, "М")
    # Конструкция assert используется для проверки истинности выражения.
    # Если выражение ложно, возбуждается исключение AssertionError.
    assert record1 == (1, "Иван", "Иванов", 20, "М")
    assert len(Student) == 1

    # Тест 2: Попытка создать запись с отрицательным возрастом.
    # Помимо положительных сценариев, важно также проверять,
    # что функция корректно обрабатывает некорректные входные данные.
    try:
        create_record(2, "Петр", "Петров", -5, "М")
        # Если функция не возбуждает ожидаемое исключение, тест считается
        # проваленным.
        # Второй аргумент функции assert - это сообщение, которое будет
        # отображено при провале теста,
        # что помогает быстрее понять причину ошибки.
        assert False, "Ожидалось исключение ValueError для отрицательного возраста."
    except ValueError as e:
        assert str(e) == "Поле age не может быть отрицательным."

    # Тест 3: Попытка создать запись с дублирующим идентификатором.
    try:
        create_record(1, "Сергей", "Сергеев", 25, "М")
        assert False, "Ожидалось исключение ValueError для дублирующего идентификатора."
    except ValueError as e:
        assert str(e) == "Запись с id=1 уже существует."
```

Приведенный пример показывает, что при тестировании следует охватывать все существенные ветви поведения подпрограммы. С этим связано понятие цикломатической сложности кода: каждое ветвление увеличивает счетчик сложности на единицу. В практическом смысле цикломатическая сложность позволяет оценить количество тестов, необходимых для полного покрытия функции.

### Организация тестов

Тесты не следует запускать путем непосредственного включения их вызовов в основной код программы. На практике они размещаются в отдельном каталоге `tests`, а для каждого тестируемого модуля создается отдельный файл с тестами. В рассматриваемом примере таким файлом является `test_memory.py`. Кроме того, рекомендуется выносить каждый тестовый случай в отдельную функцию, чтобы по результатам запуска было сразу понятно, какой именно сценарий завершился неуспешно.

Файл `tests/test_memory.py`:

```python
from src.db.backend.memory import Student, create_record


def test_create_record_positive():

    assert record1 == (1, "Иван", "Иванов", 20, "М")
    assert len(Student) == 1

    record2 = create_record(2, "Петр", "Петров", 25, "М")

    assert len(Student) == 2

    create_record(1, "Иван", "Иванов", 20, "М")
    try:
        create_record(1, "Петр", "Петров", 25, "М")

if __name__ == "__main__":
```

### Использование `unittest`

На практике для тестирования обычно применяются специализированные фреймворки. Один из них, `unittest`, входит в стандартную библиотеку Python. Основная идея состоит в том, что тесты оформляются как методы внутри классов.

```python
import unittest
from src.db.backend.memory import Student, create_record

# Класс для тестирования модуля memory.py
# Обязательное наследование от unittest.TestCase для использования функционала тестирования.
class TestMemoryBackend(unittest.TestCase):
    # В тестах есть такое понятие как fixture (фикстура) — это
    # подготовительные действия, которые выполняются перед каждым тестом.
    # Например setUp() — это метод, который вызывается перед каждым тестом
    # для подготовки тестовой среды.
    def setUp(self):
        # Очистка таблицы перед каждым тестом.
        Student.clear()

    # Тесты записываются в виде методов, имена которых начинаются с "test_".
    def test_create_record_positive_1(self):
        # Тест 1: Создание валидной записи.
        record1 = create_record(1, "Иван", "Иванов", 20, "М")
        # Вместо assert record1 == (1, "Иван", "Иванов", 20, "М") используем
        # метод assertEqual
        # для более информативного вывода в случае ошибки.
        self.assertEqual(record1, (1, "Иван", "Иванов", 20, "М"))
        self.assertEqual(len(Student), 1)

    def test_create_record_positive_2(self):
        # Тест 2: Создание ещё одной валидной записи.
        record2 = create_record(2, "Петр", "Петров", 25, "М")
        self.assertEqual(record2, (2, "Петр", "Петров", 25, "М"))
        self.assertEqual(len(Student), 2) 

    def test_create_record_age_negative(self):
        # Тест: Попытка создать запись с отрицательным возрастом.
        # Метод assertRaises используется для проверки, что при выполнении
        # кода внутри блока
        # возникает исключение указанного типа (в данном случае ValueError).
        # with это менеджер контекста, он вызывает метод __enter__() при входе
        # в блок и __exit__() при выходе из блока, что позволяет корректно
        # обрабатывать
        # исключения и другие ресурсы.
        with self.assertRaises(ValueError) as context:
            create_record(1, "Иван", "Иванов", -5, "М")
        # Метод assertEqual используется для сравнения строки исключения с
        # ожидаемым сообщением.    
        self.assertEqual(str(context.exception), "Поле age не может быть отрицательным.")

    def test_create_record_duplicate_id(self):
        # Тест: Попытка создать запись с дублирующим идентификатором.
        create_record(1, "Иван", "Иванов", 20, "М")
        with self.assertRaises(ValueError) as context:
            create_record(1, "Петр", "Петров", 25, "М")
        self.assertEqual(str(context.exception), "Запись с id=1 уже существует.")

```

В этом случае конструкция с `__main__` больше не требуется. Однако необходимо создать файл `__init__`, чтобы обозначить каталог с тестами как отдельный пакет. Для запуска тестов используется команда `python -m unittest`. Эта команда ищет файлы с префиксом `test_`, находит в них классы с именем `Test` и выполняет методы с префиксом `test_`.

### Использование `pytest`

Несмотря на удобство `unittest`, промышленным стандартом во многих проектах считается пакет `pytest`, который устанавливается отдельно командой `pip install pytest`. В отличие от `unittest`, этот фреймворк не требует столь жесткой структуры и запускает тесты во всех сущностях, имена которых начинаются с `test`.

```python
# импортируем pytest для тестирования исключений
import pytest

from src.db.backend.memory import Student, create_record

# Функция setup_function выполняется перед каждым тестом,
# обеспечивая чистое состояние таблицы Student.
def setup_function():
    Student.clear()


def test_create_record_positive_1():
    record1 = create_record(1, "Иван", "Иванов", 20, "М")

    # pytest использует обычные операторы assert для проверки условий.
    assert record1 == (1, "Иван", "Иванов", 20, "М")
    assert len(Student) == 12


def test_create_record_positive_2():
    record2 = create_record(2, "Петр", "Петров", 25, "М")

    assert record2 == (2, "Петр", "Петров", 25, "М")
    assert len(Student) == 1


def test_create_record_age_negative():
    # raises используется для проверки, что при
    # выполнении кода внутри блока возникает исключение указанного типа.
    with pytest.raises(ValueError) as exc_info:
        create_record(1, "Иван", "Иванов", -5, "М")

    assert str(exc_info.value) == "Поле age не может быть отрицательным."


def test_create_record_duplicate_id():
    create_record(1, "Иван", "Иванов", 20, "М")

    with pytest.raises(ValueError) as exc_info:
        create_record(1, "Петр", "Петров", 25, "М")

    assert str(exc_info.value) == "Запись с id=1 уже существует."

```

### Покрытие кода тестами

Дополнительно используется понятие покрытия кода тестами. Для его вычисления применяются сторонние библиотеки, например `coverage`, устанавливаемая командой `pip install coverage`, с последующим запуском `coverage run`. Аналогичную задачу для `pytest` решает плагин `pytest-cov`, устанавливаемый командой `pip install pytest-cov` и запускаемый, например, через `pytest --cov`.

## TDD на примере стека

### Общая постановка задачи

Из практики тестирования вырастают отдельные методологии разработки. Одна из них, TDD (`Test-Driven Development`), предполагает, что сначала формулируется неуспешный тест, а затем создается реализация, обеспечивающая его прохождение.

Рассмотрим данный подход на примере структуры данных стек.

## Стек

Стек рассматривался ранее при обсуждении стека вызовов функций. Это структура данных типа LIFO (`Last In, First Out`), то есть первым извлекается элемент, добавленный последним. Стек часто реализуется на основе массива или списка.

Предположим, что стек должен поддерживать следующие методы:

- `push(item)` — добавление элемента
- `pop()` — удаление элемента
- `peek()` — просмотр верхнего элемента без удаления
- `is_empty()` — проверка на пустоту
- `size()` — получение количества элементов

### Первая итерация

На первой итерации целесообразно начать с простейшей проверки метода `is_empty()`.

```python
# test_stack.py
def test_new_stack_is_empty():
    stack = Stack()
    assert stack.is_empty() is True
```

Следующим шагом является реализация метода `push`. Сначала создается тест, проверяющий, что после добавления элемента стек перестает быть пустым.

### Реализация `peek`

Завершающим шагом является добавление метода `peek()`, возвращающего верхний элемент без удаления.

```python
def test_peek_returns_top_item_without_removing_it():
    stack = Stack()
    stack.push(10)
    stack.push(20)

    result = stack.peek()

    assert result == 20
    assert stack.size() == 2
    
def test_peek_from_empty_stack_raises_index_error():
    stack = Stack()

    with pytest.raises(IndexError):
        stack.peek()    
```

```python
class Stack:
    def __init__(self):
        self._items = []

    def push(self, item):
        self._items.append(item)

    def pop(self):
        if self.is_empty():
            raise IndexError("pop from empty stack")
        return self._items.pop()

    def peek(self):
        if self.is_empty():
            raise IndexError("peek from empty stack")
        return self._items[-1]

    def is_empty(self):
        return len(self._items) == 0

    def size(self):
        return len(self._items)
```
