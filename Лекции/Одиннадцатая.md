# Коллизии, рехеширование и строковые хеш-функции

В прошлой лекции мы начали разговор про ассоциативные массивы и хеш-таблицы. Мы увидели, что хеш-таблица позволяет хранить пары "ключ - значение" и быстро получать значение по ключу.

Основная идея такая:

1. По ключу вычисляется хеш.
2. Хеш преобразуется в номер корзины.
3. В нужной корзине выполняется чтение или запись значения.

Но у этой схемы есть важная особенность: разные ключи могут попасть в одну и ту же корзину. Такая ситуация называется коллизией.

Сегодня разберём:

- как хеш-таблицы обрабатывают коллизии
- что такое метод цепочек
- что такое открытая адресация
- зачем нужно рехеширование
- как можно хешировать строки
- как отображения помогают строить простой поисковый индекс

## Коллизии. Метод цепочек

Какую бы хеш-функцию мы ни выбрали, полностью исключить коллизии нельзя. Напомню:

Коллизия - это ситуация, когда разные ключи дают одинаковый хеш или попадают в одну и ту же корзину хеш-таблицы.

Хеш-таблицу нужно проектировать с учётом этой особенности. Один из самых простых способов обработки коллизий - метод цепочек.

### Общая идея

В методе цепочек каждая корзина хранит не одну пару "ключ - значение", а набор таких пар.

Например, внутри корзины может лежать:

- список
- связанный список
- другой небольшой контейнер

Если несколько ключей попали в одну корзину, они просто добавляются в одну цепочку.

Схематично это можно представить так:

```text
H[0] -> []
H[1] -> [("Москва", "Россия"), ("Казань", "Россия")]
H[2] -> [("Вашингтон", "США")]
H[3] -> []
```

Здесь ключи `"Москва"` и `"Казань"` попали в одну корзину, но таблица всё равно может хранить оба значения, потому что внутри корзины есть цепочка.

### Добавление элемента

Допустим, нужно добавить ключ `k`, которого ещё нет в таблице.

Алгоритм:

1. Вычисляем хеш: `x = h(k)`.
2. Получаем номер корзины: `i = x mod M`, где `M` - количество корзин.
3. Переходим к корзине `H[i]`.
4. Добавляем пару "ключ - значение" в цепочку этой корзины.

Если ключ уже есть в цепочке, новое значение должно заменить старое. Иначе в отображении появились бы две пары с одинаковым ключом, а это нарушает смысл ассоциативного массива.

### Поиск элемента

Поиск устроен похожим образом:

1. Вычисляем хеш: `x = h(k)`.
2. Получаем номер корзины: `i = x mod M`.
3. Переходим к корзине `H[i]`.
4. Последовательно просматриваем цепочку и ищем нужный ключ.

Если ключ найден, возвращаем связанное с ним значение. Если вся цепочка просмотрена, а ключа нет, значит такого элемента в таблице нет.

### Реализация

```python
class Pair:
    def __init__(self, key, value):
        self.key = key
        self.value = value


class Map:
    def __init__(self, n=100):
        if n <= 0:
            raise ValueError("Размер таблицы должен быть положительным")

        # Каждая корзина хранит цепочку пар "ключ - значение".
        self.table = [[] for _ in range(n)]

    def get_bucket(self, key):
        # Учебная хеш-функция: ключи одинаковой длины попадут
        # в одну корзину, поэтому коллизии здесь легко увидеть.
        return len(key) % len(self.table)

    def set(self, key, value):
        bucket = self.get_bucket(key)
        chain = self.table[bucket]

        # Если ключ уже есть в цепочке, обновляем его значение.
        for pair in chain:
            if pair.key == key:
                pair.value = value
                return

        # Если такого ключа ещё нет, добавляем новую пару.
        chain.append(Pair(key, value))

    def get(self, key):
        bucket = self.get_bucket(key)
        chain = self.table[bucket]

        # Внутри выбранной корзины всё равно нужен поиск по ключу.
        for pair in chain:
            if pair.key == key:
                return pair.value

        return None


if __name__ == "__main__":
    city_to_country = Map()
    city_to_country.set("Вашингтон", "США")
    city_to_country.set("Москва", "Россия")

    # У слов "Москва" и "Казань" одинаковая длина,
    # поэтому в этой учебной реализации они попадут в одну корзину.
    city_to_country.set("Казань", "Россия")

    print(city_to_country.get("Москва"))  # Россия
    print(city_to_country.get("Казань"))  # Россия
    print(city_to_country.get("Берлин"))  # None
```

Эта реализация специально использует очень простую хеш-функцию. В реальных хеш-таблицах хеш-функции устроены сложнее и стремятся распределять ключи по корзинам более равномерно.

### Асимптотика метода цепочек

В худшем случае все ключи могут попасть в одну корзину. Тогда хеш-таблица фактически превратится в обычный список пар, и операции будут работать за `O(n)`.

Но если хеш-функция распределяет ключи равномерно, средняя длина цепочки будет небольшой.

Для оценки вводят коэффициент заполнения:

```text
alpha = N / M
```

где:

- `N` - количество элементов в таблице
- `M` - количество корзин
- `alpha` - среднее количество элементов на одну корзину

Если хеш-функция равномерная, вероятность попасть в каждую корзину примерно равна `1 / M`.

Пусть в корзине с номером `i` лежит `N_i` элементов. Тогда средняя стоимость обращения к корзине и просмотра её цепочки оценивается так:

```text
(1 / M) * sum(1 + N_i) = (1 / M) * M + (1 / M) * N = 1 + N / M = 1 + alpha
```

Отсюда средняя сложность поиска, вставки и обновления:

```text
O(1 + alpha)
```

Если поддерживать `alpha` в разумных пределах, эти операции в среднем работают как `O(1)`.

## Коллизии. Метод открытой адресации

Метод цепочек прост и удобен, но у него есть недостатки:

- для элементов цепочек могут потребоваться дополнительные выделения памяти
- данные в цепочках не всегда лежат последовательно в памяти
- проход по длинной цепочке может быть медленным

Другой подход к обработке коллизий называется открытой адресацией.

### Общая идея

В открытой адресации все пары "ключ - значение" хранятся прямо в массиве хеш-таблицы. Отдельных цепочек нет.

Если нужная корзина уже занята, алгоритм ищет другую корзину по некоторому правилу. Последовательность проверяемых корзин называется последовательностью проб, а способ её построения - стратегией пробирования.

Самая простая стратегия - линейное пробирование.

Формула:

```text
h(k, i) = (h(k) + i) mod M
```

где:

- `h(k)` - хеш ключа
- `i` - номер попытки
- `M` - количество корзин

То есть сначала проверяется исходная корзина, затем следующая, затем ещё следующая, и так далее. Если алгоритм доходит до конца массива, он продолжает поиск с начала таблицы.

### Добавление элемента

Алгоритм добавления при линейном пробировании:

1. Вычисляем начальный индекс: `x = h(k) mod M`.
2. Если корзина `H[x]` свободна, записываем туда пару.
3. Если корзина занята другим ключом, проверяем следующую корзину.
4. Продолжаем поиск свободной корзины циклически по массиву.
5. Если найден тот же ключ, обновляем значение.
6. Если свободной корзины нет, таблица заполнена.

### Поиск элемента

Алгоритм поиска:

1. Вычисляем начальный индекс: `x = h(k) mod M`.
2. Проверяем корзину `H[x]`.
3. Если в ней лежит нужный ключ, возвращаем значение.
4. Если корзина пустая, поиск можно остановить: такого ключа в таблице нет.
5. Если корзина занята другим ключом, проверяем следующую корзину.

Важно, что поиск идёт по той же последовательности корзин, по которой выполнялась вставка.

### Реализация

```python
class Pair:
    def __init__(self, key, value):
        self.key = key
        self.value = value


class Map:
    def __init__(self, n=100):
        if n <= 0:
            raise ValueError("Размер таблицы должен быть положительным")

        # None означает, что корзина свободна.
        self.table = [None] * n

    def get_bucket(self, key):
        # Учебная хеш-функция, специально допускающая коллизии.
        return len(key) % len(self.table)

    def set(self, key, value):
        bucket = self.get_bucket(key)

        # Проверяем не больше len(self.table) корзин,
        # чтобы не уйти в бесконечный цикл при заполненной таблице.
        for step in range(len(self.table)):
            index = (bucket + step) % len(self.table)
            pair = self.table[index]

            if pair is None:
                self.table[index] = Pair(key, value)
                return

            if pair.key == key:
                pair.value = value
                return

        raise RuntimeError("Таблица заполнена, невозможно добавить пару")

    def get(self, key):
        bucket = self.get_bucket(key)

        for step in range(len(self.table)):
            index = (bucket + step) % len(self.table)
            pair = self.table[index]

            # Пустая корзина означает, что при вставке мы бы остановились здесь.
            # Значит, дальше искомого ключа быть не может.
            if pair is None:
                return None

            if pair.key == key:
                return pair.value

        return None


if __name__ == "__main__":
    city_to_country = Map()
    city_to_country.set("Вашингтон", "США")
    city_to_country.set("Москва", "Россия")

    # Коллизия разрешается поиском следующей свободной корзины.
    city_to_country.set("Казань", "Россия")

    print(city_to_country.get("Москва"))  # Россия
    print(city_to_country.get("Казань"))  # Россия
    print(city_to_country.get("Берлин"))  # None
```

### Проблема удаления

В открытой адресации нельзя просто заменить удаляемый элемент на `None`.

Представим, что несколько элементов были записаны подряд из-за коллизий. Если удалить первый из них и сделать корзину пустой, поиск следующих элементов может остановиться слишком рано.

Поэтому в таких таблицах часто используют специальную пометку "элемент удалён". Такая корзина:

- считается занятой для поиска
- может быть переиспользована при новой вставке

### Кластеры

Если хеш-функция распределяет значения неравномерно, в таблице могут появляться длинные группы занятых корзин. Такие группы называют кластерами.

Кластеры ухудшают производительность, потому что при поиске и вставке приходится просматривать больше корзин.

Один из способов уменьшить эту проблему - использовать более сложные стратегии пробирования.

## Стратегии пробирования

### Линейное пробирование с шагом

Обычное линейное пробирование проверяет соседние корзины. Можно обобщить его и использовать шаг `c`:

```text
h(k, i) = (h(k) + c * i) mod M
```

Чтобы алгоритм мог пройти по всей таблице, числа `M` и `c` должны быть взаимно простыми, то есть не иметь общих делителей, кроме `1`.

Если у `M` и `c` есть общий делитель, алгоритм будет проверять только часть корзин, а до остальных никогда не дойдёт.

Например, если `M` - степень двойки, то удобно выбирать нечётный шаг `c`. У степени двойки единственный простой делитель - `2`, поэтому нечётное число будет с ней взаимно простым.

### Квадратичное пробирование

Ещё один вариант - квадратичное пробирование:

```text
h(k, i) = (h(k) + c1 * i + c2 * i^2) mod M
```

Здесь расстояние между проверяемыми корзинами растёт не линейно, а по квадратичному закону. Это может уменьшить образование больших кластеров, характерных для обычного линейного пробирования.

### Двойное хеширование

При двойном хешировании используются две хеш-функции:

```text
h(k, i) = (h1(k) + i * h2(k)) mod M
```

Первая хеш-функция выбирает начальную корзину, а вторая задаёт шаг пробирования.

Для корректной работы важно, чтобы `h2(k)` был взаимно прост с `M`. Если `M` является степенью двойки, часто делают так, чтобы `h2(k)` всегда возвращал нечётное число.

## Рехеширование

Размер хеш-таблицы должен соответствовать количеству элементов.

Если таблица слишком большая, память используется неэффективно. Если таблица слишком маленькая, возникает много коллизий, цепочки становятся длинными, а при открытой адресации растёт число проб.

Поэтому хеш-таблицы обычно изменяют размер динамически. Такой процесс называется рехешированием.

### Когда нужно менять размер таблицы

Ориентируются на коэффициент заполнения:

```text
alpha = N / M
```

Если `alpha` становится слишком большим, таблицу увеличивают. Если после большого количества удалений `alpha` становится слишком маленьким, таблицу иногда уменьшают.

Оптимальные границы зависят от конкретной реализации:

- для метода цепочек `alpha` может быть около `1` или даже больше
- для открытой адресации `alpha` должен быть меньше `1`, иначе свободные корзины закончатся
- многие промышленные реализации используют пороги примерно от `0.5` до `0.75`

### Алгоритм рехеширования

Общий алгоритм:

1. Выбираем новый размер таблицы.
2. Создаём новый массив корзин.
3. Проходим по всем старым элементам.
4. Для каждого ключа заново вычисляем номер корзины уже по новому размеру.
5. Вставляем элемент в новую таблицу.

Важно: нельзя просто скопировать элементы в те же индексы. Номер корзины зависит от размера таблицы, поэтому после изменения `M` почти все элементы могут получить новые позиции.

Если используется метод деления, часто выбирают простые размеры таблицы:

```text
7, 17, 31, 67, 127, 257, 509, 1021, ...
```

При увеличении таблицы переходят к следующему размеру, при уменьшении - к предыдущему.

Рехеширование - дорогая операция. Её сложность:

```text
O(n)
```

Но выполняется она редко. Поэтому вставка в хеш-таблицу обычно имеет амортизированную сложность `O(1)`.

Амортизированная сложность означает, что одна конкретная операция иногда может быть дорогой, но средняя стоимость операции на длинной последовательности действий остаётся небольшой.

## Хеш-функции для строк

Теперь рассмотрим более реалистичный пример: как можно вычислять хеш для строки.

Строку можно воспринимать как последовательность чисел, где каждому символу соответствует числовой код. Например, в Python код символа можно получить функцией `ord()`.

Один из классических способов хеширования строк - полиномиальное хеширование.

Для строки длины `n` формула выглядит так:

```text
h(s) = (s1 * q^(n - 1) + s2 * q^(n - 2) + ... + s(n - 1) * q + sn) mod R
```

где:

- `s1, s2, ..., sn` - числовые коды символов строки
- `q` - выбранное основание
- `R` - модуль, ограничивающий размер результата
- `n` - длина строки

Модуль нужен потому, что для длинных строк значение полинома быстро становится очень большим.

В учебных и алгоритмических задачах часто выбирают большое простое `R`. В системных реализациях также могут использовать переполнение машинного целого, что фактически похоже на вычисления по модулю `2^32` или `2^64`.

Основание `q` обычно выбирают так, чтобы оно помогало хорошо перемешивать символы. Для учебных примеров можно брать маленькие числа, но в реальных задачах используют более аккуратно подобранные константы.

### Прямое вычисление полинома

```python
# Маленькие константы выбраны только для наглядности примера.
Q = 3
R = 256


def string_hash(s: str) -> int:
    h = 0

    for i, char in enumerate(s):
        power = len(s) - i - 1

        # Каждый символ даёт очередное слагаемое полинома.
        # Остаток по модулю R удерживает значение хеша в заданном диапазоне.
        h = (h + ord(char) * Q ** power) % R

    return h


print(string_hash("ABACB"))  # 219
```

Такой вариант хорошо показывает формулу, но он не самый удобный: приходится явно вычислять степени `Q`.

### Метод Горнера

Полиномиальный хеш можно вычислять эффективнее с помощью метода Горнера.

Та же формула переписывается так:

```text
h(s) = (((s1 * q + s2) * q + s3) * q + ... + sn) mod R
```

Реализация становится проще:

```python
# Те же константы, что и в предыдущем примере.
Q = 3
R = 256


def string_hash(s: str) -> int:
    h = 0

    for char in s:
        # Метод Горнера добавляет символы слева направо.
        # На каждом шаге предыдущий результат умножается на основание Q.
        h = (h * Q + ord(char)) % R

    return h


print(string_hash("ABACB"))  # 219
```

Метод Горнера обычно предпочтительнее, потому что он не требует отдельно считать степени и проходит по строке один раз.

## Поисковый индекс

Теперь посмотрим, как отображения помогают искать слова в тексте.

Допустим, у нас есть строка:

```python
TEXT = "быть или не быть вот в чем вопрос"
```

Если каждый раз искать подстроку обычным проходом по тексту, это может быть дорого. Поэтому можно заранее построить индекс.

Идея индекса:

- ключом будет слово
- значением будет список позиций, с которых это слово начинается в тексте

Например:

```text
"быть" -> [0, 12]
"или" -> [5]
"не" -> [9]
```

Процесс построения такого отображения называется индексированием.

Если нужно найти одно слово, достаточно обратиться к словарю. Если нужно найти несколько слов подряд, можно взять самое редкое слово из запроса и проверить только его позиции.

### Реализация

```python
TEXT = "быть или не быть вот в чем вопрос"

search_index = {}
source_text = ""


def fill_search_index(text):
    """
    Строит индекс слов:
    ключ - слово, значение - список начальных позиций этого слова в тексте.
    """
    global source_text

    search_index.clear()
    source_text = text

    position = 0

    for word in text.split():
        if word not in search_index:
            search_index[word] = []

        search_index[word].append(position)

        # В этом учебном примере считаем, что слова разделены одним пробелом.
        position += len(word) + 1


def find_substring_positions(substring):
    """
    Возвращает позиции, с которых в исходном тексте начинается substring.
    Поиск опирается на индекс слов, а точное совпадение проверяется срезом строки.
    """
    words = substring.split()

    if not words:
        return []

    if len(words) == 1:
        return search_index.get(words[0], []).copy()

    rarest_word_index = None
    rarest_word_positions = None

    # Выбираем слово с наименьшим количеством вхождений,
    # чтобы уменьшить число точных проверок по исходному тексту.
    for index, word in enumerate(words):
        positions = search_index.get(word, [])

        if not positions:
            return []

        if rarest_word_positions is None or len(positions) < len(rarest_word_positions):
            rarest_word_index = index
            rarest_word_positions = positions

    prefix = " ".join(words[:rarest_word_index])
    start_offset = len(prefix) + 1 if prefix else 0

    positions = []

    for word_position in rarest_word_positions:
        start = word_position - start_offset
        end = start + len(substring)

        if start < 0:
            continue

        # Индекс даёт только кандидатов, а срез подтверждает точное совпадение.
        if source_text[start:end] == substring:
            positions.append(start)

    return positions


if __name__ == "__main__":
    fill_search_index(TEXT)

    print(search_index)
    print(find_substring_positions("быть"))      # [0, 12]
    print(find_substring_positions("не быть"))   # [9]
    print(find_substring_positions("быть вот"))  # [12]
```

Этот пример намеренно упрощён. Он корректно работает для текста, где слова разделены одним пробелом и нет знаков препинания, которые нужно обрабатывать отдельно.

В реальных поисковых системах индексирование сложнее. Обычно дополнительно выполняют:

- нормализацию регистра
- обработку пунктуации
- приведение слов к начальной форме
- хранение номеров документов
- ранжирование результатов

Но базовая идея остаётся той же: отображение позволяет быстро перейти от слова к местам, где оно встречается.
