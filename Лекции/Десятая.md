# Отображения, ассоциативные массивы и хеш-таблицы

Во многих задачах программирования нам нужно быстро получать данные и быстро записывать их. Если говорить о структурах данных, то одной из самых удобных базовых структур для такой задачи является массив.

Почему массив так удобен:

- чтение по индексу работает за `O(1)`
- запись по индексу тоже работает за `O(1)`

То есть если мы знаем индекс, то можем почти мгновенно получить или изменить значение.

Но здесь есть важное ограничение: массив работает именно с целочисленными индексами. А в реальных задачах нам часто нужно получать данные не по числу, а по какому-то другому ключу:

- по строке
- по идентификатору
- по названию города
- по имени пользователя

Чтобы понять, как такие структуры устроены, сначала разберём само понятие отображения.

## Отображение

Говоря про массив, мы можем сказать, что каждому целочисленному индексу соответствует одно значение. Когда одному значению ставится в соответствие другое значение, это называют отображением.

Например:

- одному значению города соответствует одна страна
- одному номеру месяца соответствует его название
- одному идентификатору пользователя соответствует его запись в системе

Если говорить в терминах математики, отображение часто записывают как функцию:

$$
f(x) = y
$$

Это означает, что каждому значению `x` поставлено в соответствие некоторое значение `y`.

При этом важно:

- одному `x` соответствует ровно одно `y`
- одному `y` могут соответствовать несколько разных `x`

Например, нескольким городам может соответствовать одна и та же страна.

С этой точки зрения массив - это частный случай отображения:

```text
[0] -> "Значение 1"
[1] -> "Значение 2"
[2] -> "Значение 3"
```

Работу с массивом можно описать таким интерфейсом:

```text
get(index: int) -> Any
set(index: int, value: Any)
```

Но очень часто нам нужно получать значение не по числу, а, например, по строке.

Например:

```text
["Москва"] -> "Россия"
["Вашингтон"] -> "США"
["Казань"] -> "Россия"
```

То есть нам нужен уже такой интерфейс:

```text
get(key: KeyType) -> Any
set(key: KeyType, value: Any)
```

Такой интерфейс обычно называют `Map` (`Отображение`), а структуры данных, которые его реализуют, называют ассоциативными массивами.

## Ассоциативный массив

Ассоциативный массив - это структура данных, которая хранит пары:

- ключ
- значение

и позволяет быстро получать значение по ключу.

В разных языках программирования эта структура называется по-разному:

- `map`
- `hash map`
- `hash table`
- `dictionary`

В Python такой структурой являются словари `dict`.

### Простейший пример

```python
country_by_city = {
    "Москва": "Россия",
    "Вашингтон": "США",
    "Казань": "Россия",
}

print(country_by_city["Москва"])  # Россия
print(country_by_city["Вашингтон"])  # США
```

Здесь:

- ключами являются названия городов
- значениями являются названия стран

### Пример с месяцами

Допустим, нам нужно по названию месяца получать его номер.

```python
month_number = {
    "Январь": 1,
    "Февраль": 2,
    "Март": 3,
    "Апрель": 4,
    "Май": 5,
    "Июнь": 6,
    "Июль": 7,
    "Август": 8,
    "Сентябрь": 9,
    "Октябрь": 10,
    "Ноябрь": 11,
    "Декабрь": 12,
}

# В среднем обращение по ключу работает за O(1).
print(month_number["Март"])  # 3
print(month_number["Ноябрь"])  # 11
```

### Пример с подсчётом

Ассоциативные массивы очень часто используются для подсчёта количества объектов.

```python
fruits = ["яблоко", "банан", "яблоко", "апельсин", "яблоко", "банан"]

fruits_count = {}

for fruit in fruits:
    if fruit in fruits_count:
        # Если ключ уже есть, увеличиваем счётчик.
        fruits_count[fruit] += 1
    else:
        # Если ключа ещё нет, создаём его со значением 1.
        fruits_count[fruit] = 1

print(fruits_count)
# {'яблоко': 3, 'банан': 2, 'апельсин': 1}
```

Это очень типичная задача. Именно из-за подобных сценариев словари используются постоянно.

## Кэширование с помощью словаря

Ассоциативные массивы очень часто применяются для кэширования.

Идея кэша простая:

- если мы уже один раз получили какие-то данные
- то не нужно получать их заново из медленного источника
- можно сохранить результат в словарь и использовать повторно

Это особенно полезно, когда источник данных медленный:

- база данных
- файл
- сеть
- внешний API

Представим, что у нас есть данные такого вида:

```json
{
    "data": [
        {
            "student_id": 0,
            "group_id": 12554
        },
        {
            "student_id": 1,
            "group_id": 12354
        },
        {
            "student_id": 2,
            "group_id": 12554
        }
    ]
}
```

Допустим, чтобы получить подробную информацию о студенте и его группе, нам приходится обращаться к базе данных. Такие обращения - это `I/O`-операции, а `I/O`-операции обычно дорогие по времени.

Если одна и та же группа встречается много раз, не хочется каждый раз заново делать запрос в базу. Для этого и нужен кэш.

```python
data = [
    {"student_id": 0, "group_id": 12554},
    {"student_id": 1, "group_id": 12354},
    {"student_id": 2, "group_id": 12554},
]


def student_by_id(student_id):
    """
    Примерная функция получения студента по его идентификатору.
    Представим, что внутри здесь выполняется медленный запрос к базе данных.
    """
    return {"student_id": student_id, "name": f"Student {student_id}"}


def students_by_group(group_id):
    """
    Примерная функция получения всех студентов группы.
    Представим, что здесь тоже выполняется медленный запрос к базе данных.
    """
    return [f"student_from_group_{group_id}_1", f"student_from_group_{group_id}_2"]


def main():
    # Здесь будем хранить уже загруженные данные по группам.
    cache = {}

    for item in data:
        student_id = item["student_id"]
        group_id = item["group_id"]

        # Данную информацию мы пока не кэшируем,
        # но при желании могли бы кэшировать и её.
        student_data = student_by_id(student_id)

        # Если группы ещё нет в кэше, загружаем её из "базы".
        if group_id not in cache:
            cache[group_id] = students_by_group(group_id)

        # Если группа уже была загружена ранее,
        # повторный запрос не нужен.
        group_data = cache[group_id]

        print(student_data)
        print(group_data)
        print()
        # Для student_id = 0 вывод будет примерно таким:
        # {'student_id': 0, 'name': 'Student 0'}
        # ['student_from_group_12554_1', 'student_from_group_12554_2']
        #
        # Для student_id = 1:
        # {'student_id': 1, 'name': 'Student 1'}
        # ['student_from_group_12354_1', 'student_from_group_12354_2']
        #
        # Для student_id = 2:
        # {'student_id': 2, 'name': 'Student 2'}
        # ['student_from_group_12554_1', 'student_from_group_12554_2']
        #
        # При этом для group_id = 12554 запрос к "базе" выполняется только один раз.


if __name__ == "__main__":
    main()
```

Словарь здесь работает как очень простой и очень полезный кэш.

## Попробуем реализовать ассоциативный массив сами

Прежде чем переходить к хеш-таблицам, давайте попробуем реализовать собственный ассоциативный массив "в лоб".

```python
class Pair:
    def __init__(self, key, value):
        self.key = key
        self.value = value


class Map:
    def __init__(self):
        self.pairs = []

    def get(self, key):
        for pair in self.pairs:
            if pair.key == key:
                return pair.value

        # Если ключ не найден, возвращаем None.
        return None

    def set(self, key, value):
        for pair in self.pairs:
            if pair.key == key:
                pair.value = value
                return

        # Если ключа ещё нет, добавляем новую пару.
        self.pairs.append(Pair(key, value))


city_to_country = Map()
city_to_country.set("Москва", "Россия")
city_to_country.set("Вашингтон", "США")
city_to_country.set("Москва", "Российская Федерация")

print(city_to_country.get("Москва"))  # Российская Федерация
print(city_to_country.get("Казань"))  # None
```

### В чём проблема такой реализации

Эта реализация корректна по смыслу, но не очень эффективна.

Почему:

- для `get()` приходится последовательно просматривать весь список пар
- для `set()` тоже приходится искать, есть ли уже такой ключ

Значит, и получение, и запись здесь занимают:

```text
O(n)
```

То есть если элементов становится много, такая структура начинает работать заметно хуже, чем нам хотелось бы.

Именно здесь на сцену выходят хеш-таблицы.

## Хеш-таблицы и хеш-функции

### Как ускорить ассоциативный массив

Мы уже знаем, что массив очень хорош тем, что доступ по индексу в нём занимает `O(1)`. Значит, хочется придумать способ переводить произвольный ключ в индекс массива.

Если бы мы умели это делать быстро, то могли бы:

1. по ключу вычислить индекс
2. перейти в нужную ячейку массива
3. получить или записать значение

Именно на этой идее основаны хеш-таблицы.

### Наивный пример

Пусть у нас есть фрукты:

- яблоко
- слива
- груша

И мы заранее договорились:

- `"яблоко"` будет лежать в корзине `0`
- `"слива"` - в корзине `1`
- `"груша"` - в корзине `2`

Тогда можно написать такую функцию:

```python
def bucket(fruit):
    if fruit == "яблоко":
        return 0
    elif fruit == "слива":
        return 1
    elif fruit == "груша":
        return 2
    else:
        raise ValueError("Неизвестный фрукт")
```

А сама таблица будет выглядеть так:

```python
hash_table = [("яблоко", 11), ("слива", 20), ("груша", 8)]
```

Тогда, чтобы узнать количество слив, мы делаем так:

```python
def bucket(fruit):
    if fruit == "яблоко":
        return 0
    elif fruit == "слива":
        return 1
    elif fruit == "груша":
        return 2
    else:
        raise ValueError("Неизвестный фрукт")


hash_table = [("яблоко", 11), ("слива", 20), ("груша", 8)]

print(hash_table[bucket("слива")][1])  # 20
```

Идея уже похожа на хеш-таблицу:

- данные хранятся в массиве
- по ключу вычисляется номер корзины (`bucket`)
- из корзины получаем значение

Но это ещё очень упрощённая схема, потому что:

- набор ключей заранее известен
- функция написана вручную
- не описана обработка коллизий

Тем не менее, как учебная модель эта идея полезна.

### Что такое хеш-функция

Функция `bucket()` в примере выше - это наивная версия хеш-функции.

Хеш-функция - это функция, которая преобразует входные данные в некоторое число. Результат её работы называют:

- хешем
- хеш-значением
- хеш-суммой

В контексте хеш-таблиц это число потом превращается в индекс корзины.

## Коллизии

Теперь представим, что мы хотим придумать более общую хеш-функцию. Например, пусть она возвращает номер первой буквы слова:

- для `"яблоко"` это будет один номер
- для `"слива"` другой
- для `"груша"` третий

Но что произойдёт, если у нас есть:

- `"арбуз"`
- `"абрикос"`

У них первая буква одинаковая. Значит, обе строки попадут в одну и ту же корзину.

Такое явление называется коллизией.

Коллизия - это ситуация, когда разные ключи дают один и тот же результат хеш-функции или попадают в одну и ту же корзину.

Коллизии в реальных хеш-таблицах неизбежны. Важно не избежать их полностью, а уметь корректно с ними работать.

## Более реалистичная схема

Можно попробовать вычислять хеш более хитро, например не по первой букве, а как сумму кодов символов.

Пусть:

- для `"яблоко"` хеш равен `92`
- для `"груша"` хеш равен `70`
- для `"слива"` хеш равен `46`

Но хранить массив длины `92` только ради нескольких фруктов невыгодно. Поэтому в реальных хеш-таблицах обычно делают два шага:

1. Сначала вычисляют хеш.
2. Потом преобразуют хеш в номер корзины.

Например, если корзин `8`, то можно сделать так:

```python
92 % 8 = 4
```

То есть:

1. Вычислили хеш `92`.
2. Взяли остаток от деления на `8`.
3. Получили индекс корзины `4`.
4. Сохранили значение в этой корзине.

В реальных таблицах в одной корзине может храниться несколько значений, поэтому внутри корзины ещё нужен поиск.

## Выбор размера хеш-таблицы и вычисление корзины

Чтобы упростить рассуждения, дальше будем считать, что ключи у нас целочисленные.

Можно подумать: если ключи целочисленные, то зачем вообще хеш-таблица, ведь можно просто использовать массив. Но тут есть две проблемы:

1. Ключи могут быть отрицательными.
2. Ключи могут быть очень разреженными.

Например, если у нас есть ключи:

```python
1, 55, 1000, 9999999
```

то использовать обычный массив неудобно. Пришлось бы заводить огромный массив ради нескольких значений.

Для простоты возьмём такую хеш-функцию:

```python
def hash_value(key: int) -> int:
    return key
```

То есть пока будем считать, что хеш просто равен самому ключу.

### Метод деления

Один из самых простых способов получить номер корзины - взять остаток от деления хеша на количество корзин.

```python
def bucket(hash_value: int, m: int) -> int:
    return hash_value % m


print(bucket(118, 5))  # 3
```

Здесь:

- `118` - хеш
- `5` - количество корзин
- `3` - номер корзины

Потому что:

```python
118 % 5 = 3
```

## Почему число корзин важно

Если корзин слишком мало, будет слишком много коллизий.
Если корзин слишком много, мы зря потратим память на пустые корзины.

Поэтому размер таблицы важен.

### Почему часто выбирают простые числа

На практике часто рекомендуют выбирать число корзин простым. Давайте посмотрим, почему.

Пусть размер таблицы равен `6`, а ключи образуют арифметическую прогрессию с шагом `4`:

```python
0, 4, 8, 12, ..., 104
```

Если посчитать остатки по модулю `6`, получим:

```text
[0]: 0, 12, 24, 36, 48, 60, 72, 84, 96
[2]: 8, 20, 32, 44, 56, 68, 80, 92, 104
[4]: 4, 16, 28, 40, 52, 64, 76, 88, 100
```

Итого:

- используются только корзины `0`, `2` и `4`
- половина таблицы остаётся пустой

Это происходит потому, что шаг прогрессии и число корзин имеют общий делитель.

Теперь уменьшим число корзин до `5`:

```text
[0]: 0, 20, 40, 60, 80, 100
[1]: 16, 36, 56, 76, 96
[2]: 12, 32, 52, 72, 92
[3]: 8, 28, 48, 68, 88
[4]: 4, 24, 44, 64, 84, 104
```

Теперь распределение уже заметно равномернее.

Конечно, и для простого числа можно придумать плохую последовательность ключей. Но в среднем такой выбор часто снижает вероятность неудачного распределения.

Итоговая функция вычисления корзины методом деления выглядит так:

```python
def bucket(hash_value: int, m: int) -> int:
    return hash_value % m
```

## Метод умножения

Есть и другой популярный способ вычисления номера корзины - метод умножения.

Пусть:

- количество корзин равно `m`
- есть константа `a`, где `0 < a < 1`

Тогда номер корзины можно вычислять по формуле:

$$
bucket(h) = \lfloor m \cdot \{h \cdot a\} \rfloor
$$

где:

- `[x]` - целая часть числа
- `{x}` - дробная часть числа

Реализация может выглядеть так:

```python
def bucket(h: int, m: int) -> int:
    # Константа, близкая к обратному золотому сечению.
    # Её часто упоминают в связи с хорошим распределением значений.
    a = 0.6180339887

    # Вычисляем дробную часть от h * a.
    d = h * a
    d = d - int(d)

    # Масштабируем результат до диапазона [0, m).
    r = m * d

    # Отбрасываем дробную часть и получаем индекс корзины.
    return int(r)


print(bucket(118, 5))  # Например: 4
```

Такой подход не зависит напрямую от деления на число корзин, но использует операции с плавающей точкой, поэтому обычно работает чуть медленнее.

В учебном курсе нам достаточно понимать саму идею. Оптимизации этого метода можно изучить отдельно.

## Свойства хеш-функций

Чтобы хеш-таблица работала хорошо, хеш-функция должна обладать несколькими важными свойствами.

### 1. Детерминизм

Для одного и того же входного значения хеш-функция всегда должна возвращать один и тот же результат.

Если этого свойства нет, то:

- сегодня запись попадёт в одну корзину
- завтра в другую

и структура перестанет работать корректно.

### 2. Эффективность

Хеш-функция должна вычисляться быстро. Нет смысла ускорять структуру данных, если само вычисление хеша будет слишком дорогим.

### 3. Равномерность

Хорошая хеш-функция должна стараться распределять значения по корзинам равномерно. Если почти все ключи попадают в несколько корзин, таблица начинает деградировать и работать всё хуже.

## Где ещё используются хеши

Хеш-функции нужны не только для хеш-таблиц.

### Проверка содержимого

Например, хеш можно использовать для проверки, совпадает ли некоторое содержимое с уже известным образцом.

Упрощённый жизненный пример:

- система заранее знает хеш некоторого запрещённого файла
- пользователь пытается отправить файл
- система вычисляет его хеш
- если хеш совпал, отправку можно заблокировать

В реальности такие системы обычно устроены сложнее, потому что даже маленькое изменение файла меняет хеш.

### Защита от повторной обработки

Ещё один практический пример - борьба с дублирующими сообщениями.

Представим:

- пользователь нажимает кнопку "Сохранить" много раз подряд
- система каждый раз формирует одно и то же JSON-сообщение
- это сообщение отправляется в очередь

Если все такие сообщения обрабатывать заново, это создаёт лишнюю нагрузку. Возможное решение:

1. Вычислить хеш сообщения.
2. Сохранить его в отдельной таблице.
3. Перед повторной отправкой проверить, встречался ли уже такой хеш.

Если хеш уже был, можно не отправлять сообщение повторно.

## Хеширование паролей

Очень важное применение хеш-функций - хранение паролей.

Когда пользователь регистрируется в сервисе, система не должна сохранять пароль в открытом виде. Вместо этого:

1. пароль хешируется
2. в базу данных записывается не сам пароль, а его хеш

На практике обычно хранят не просто хеш, а хеш вместе с солью (`salt`), но пока нам важна сама общая идея.

Когда пользователь пытается войти в систему:

1. он вводит пароль
2. система снова вычисляет его хеш
3. сравнивает новый хеш с тем, который хранится в базе

Если они совпали, пароль считается правильным.

Благодаря этому сервис не хранит пароли в открытом виде. Даже если базу данных украдут, злоумышленник не увидит исходные пароли напрямую.

Чтобы такие сценарии работали, для криптографических хеш-функций особенно важны ещё два свойства.

### 4. Лавинный эффект

Если немного изменить входные данные, хеш должен измениться очень сильно.

Например, строки:

- `"password123"`
- `"password124"`

должны давать совершенно разные хеши.

Это усложняет подбор исходного значения по похожим вариантам.

### 5. Необратимость

По хешу должно быть практически невозможно восстановить исходное значение.

Именно это и делает хеширование полезным для хранения паролей.

Важно только понимать, что для паролей нужны не "любые" хеш-функции, а специальные криптографические алгоритмы. Обычные быстрые хеши для таблиц здесь не подходят.

## Итог

В этой лекции мы разобрали несколько очень важных идей.

1. Отображение - это соответствие, при котором одному значению ключа ставится в соответствие одно значение результата.
2. Ассоциативный массив позволяет получать значение не по числовому индексу, а по произвольному ключу.
3. В Python ассоциативный массив реализован как словарь `dict`.
4. Наивная реализация ассоциативного массива через список пар работает за `O(n)` и плохо масштабируется.
5. Хеш-таблица ускоряет такую структуру за счёт преобразования ключа в индекс массива.
6. Хеш-функции важны не только для таблиц, но и для кэширования, защиты от дублей, проверки содержимого и хранения паролей.

Если смотреть шире, эта лекция показывает важную инженерную мысль: удобный интерфейс структуры данных почти всегда стоит за конкретной внутренней идеей реализации. Снаружи словарь выглядит очень просто: "дай значение по ключу". Но внутри за этим стоят отображения, хеш-функции, корзины, коллизии и компромиссы между скоростью и памятью.
