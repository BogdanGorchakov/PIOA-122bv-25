## Введение

В рамках данной лабораторной работы рассматривается развитие in-memory базы данных, реализованной ранее, с использованием принципов объектно-ориентированного программирования и базовых подходов к тестированию. Теоретические основы объектно-ориентированного программирования и тестирования уже были рассмотрены в предшествующих учебных материалах: [ООП раз](/Лекции/Вторая.md#объектно-ориентированное-программирование-ооп), [ООП два](/Лекции/Третья.md), [Тесты](/Лекции/Пятая.md#тестирование).

Цель настоящей методички состоит в демонстрации практического подхода к переработке существующего решения в объектно-ориентированную форму и в освоении базового цикла разработки через тестирование (TDD, *Test-Driven Development*). Основное внимание уделяется структуре проекта, поэтапному добавлению тестов и реализации функциональности на их основе.

## Основная часть

### Структура проекта из предыдущей лабораторной работы

Прежде всего целесообразно напомнить структуру проекта, полученную в предыдущей лабораторной работе.

```
PIOA-НомерГруппы/
│
├── src/
│ └── db/
│   └── backend/
│   │  ├── __init__.py
│   │  └── memory.py
│   │
│   ├── __init__.py
│   ├── __main__.py
│   └── tui.py
│
└── README.md
```

Модуль `memory.py` содержит реализацию in-memory базы данных. В исходном варианте база данных представлена списком кортежей, а ключевая функциональность сводится к операциям создания новой записи и выборки записей по фильтрам.

Для минимальной объектно-ориентированной переработки данной базы данных достаточно реализовать класс `StudentTable`. Более полная архитектура могла бы включать, как минимум, два класса: класс СУБД `Memory` и класс таблицы `Table`. Однако такая декомпозиция относится к самостоятельной доработке.

### Переход к объектно-ориентированной реализации и TDD

В данной лабораторной работе предлагается реализовать систему с использованием принципов TDD. Для этого необходимо расширить структуру проекта, добавив каталог для тестов `tests` и модуль тестирования `test_memory.py`. В качестве тестового фреймворка используется `unittest`.

Обновлённая структура проекта принимает следующий вид:

```
PIOA-НомерГруппы/
│
├── src/
│ └── db/
│   └── backend/
│   │  ├── __init__.py
│   │  └── memory.py
│   │
│   ├── __init__.py
│   ├── __main__.py
```

Так как используется фреймворк `unittest`, в тестовом модуле необходимо определить класс `TestMemory`, который будет содержать методы тестирования.

### Первый тест и создание класса `StudentTable`

Разработка в стиле TDD начинается с написания теста, который на первом этапе не проходит. Рассмотрим начальный пример:

```python
# tests/test_memory.py
import unittest

class TestMemory(unittest.TestCase):

    def test_student_table_allocation(self):
        student_table = StudentTable() # NameError: name 'StudentTable' is not defined
        self.assertIsInstance(student_table, StudentTable)

```

Запуск тестов осуществляется командой `python -m unittest`. Данный тест не будет выполнен по следующим причинам:

1. Класс `StudentTable` ещё не реализован.
2. Модуль с реализацией не импортирован в тестовый файл.

Для устранения первой причины добавим минимальное определение класса:

```python
# src/db/backend/memory.py
class StudentTable:
    pass
```

После этого необходимо подключить класс в тестовом модуле:

```python
# tests/test_memory.py
from src.db.backend.memory import StudentTable
```

После выполнения указанных действий тест начинает выполняться успешно.

### Тестирование и реализация метода `create_record`

Следующий этап состоит в проверке метода `create_record`. На первом шаге формулируется тест:

```python
# tests/test_memory.py
import unittest
from src.db.backend.memory import StudentTable

class TestMemory(unittest.TestCase):

    def test_student_table_allocation(self):
        student_table = StudentTable()
        self.assertIsInstance(student_table, StudentTable)

    def test_create_record(self):
        test_data = (1, "John", "Doe", 20, "M")

        student_table = StudentTable()
        record = student_table.create_record(*test_data) # AttributeError: 'StudentTable' object has no attribute 'create_record'
        self.assertEqual(record, test_data)

```

Ошибка возникает вследствие отсутствия метода `create_record`. Поэтому сначала необходимо объявить его интерфейс:

```python
# src/db/backend/memory.py
type StudentRecord = tuple[int, str, str, int, str]

class StudentTable:
    def create_record(
        self,
        student_id: int,
        first_name: str,
        second_name: str,
        age: int,
        sex: str,
    ) -> StudentRecord:
        pass
```

После добавления объявления тест завершится с ошибкой `AssertionError: None != (1, 'John', 'Doe', 20, 'M')`, поскольку метод пока не содержит реальной логики. Реализация может быть задана следующим образом:

```python
# src/db/backend/memory.py
type StudentRecord = tuple[int, str, str, int, str]


class StudentTable:
    def __init__(self) -> None:
        self._student: list[StudentRecord] = []

    def create_record(
        self,
        student_id: int,
        first_name: str,
        second_name: str,
        age: int,
        sex: str,
    ) -> StudentRecord:
        new_record: StudentRecord = (
            student_id,
            first_name.strip(),
            second_name.strip(),
            age,
            sex.strip(),
        )
        self._student.append(new_record)
        return new_record
```

После добавления данной реализации тест проходит успешно.

### Пользовательские исключения

В предыдущей лабораторной работе метод `create_record` уже содержал проверки на корректность возраста и на отсутствие дублирующихся идентификаторов. В данной версии рекомендуется оформить эти ошибки через пользовательские исключения для повышения наглядности и структурированности кода.

Для этого добавим модуль `errors.py`:

```
PIOA-НомерГруппы/
│
├── src/
│ └── db/
│   └── backend/
│   │  ├── __init__.py
|      ├── errors.py
│   │  └── memory.py
│   │
│   ├── __init__.py
│   ├── __main__.py
│   └── tui.py
│
├── tests/
| ├── __init__.py
│ └── test_memory.py
│
└── README.md
```

Для создания собственных типов ошибок требуется определить классы, наследуемые от стандартного класса `Exception`. Ниже приведён рекомендуемый пример:

```python
# src/db/backend/errors.py
class StudentTableError(Exception):
    """Базовый класс для ошибок, связанных с таблицей Student."""
    pass

class InvalidAgeError(StudentTableError):
    """Ошибка, возникающая при попытке создать запись с некорректным возрастом."""
    pass

class DuplicateIDError(StudentTableError):
    """Ошибка, возникающая при попытке создать запись с уже существующим идентификатором."""
    pass
```

После этого можно сформулировать тесты, проверяющие соответствующие исключительные ситуации:

```python
# tests/test_memory.py
import unittest
from src.db.backend.memory import StudentTable
from src.db.backend.errors import InvalidAgeError, DuplicateIDError


class TestMemory(unittest.TestCase):
    def test_student_table_allocation(self):
        student_table = StudentTable()
        self.assertIsInstance(student_table, StudentTable)

    def test_create_record(self):
        test_data = (1, "John", "Doe", 20, "M")

        student_table = StudentTable()
        record = student_table.create_record(*test_data)
        self.assertEqual(record, test_data)

    def test_create_record_negative_age(self):
        test_data = (1, "John", "Doe", -5, "M")
        error_message = "Поле age не может быть отрицательным."
        student_table = StudentTable()

        with self.assertRaises(InvalidAgeError) as context: # AssertionError: InvalidAgeError not raised
            student_table.create_record(*test_data)

        self.assertEqual(str(context.exception), error_message)

    def test_create_record_duplicate_id(self):
        test_data_1 = (1, "John", "Doe", 20, "M")
        test_data_2 = (1, "Jane", "Smith", 22, "F")
        error_message = "Запись с id=1 уже существует."
        student_table = StudentTable()

        student_table.create_record(*test_data_1)

        with self.assertRaises(DuplicateIDError) as context: # AssertionError: DuplicateIDError not raised
            student_table.create_record(*test_data_2)

        self.assertEqual(str(context.exception), error_message)    

```

На данном этапе тесты не проходят, поскольку соответствующая логика ещё не реализована. Добавим обработку ошибок в модуль `memory.py`:

```python
# src/db/backend/memory.py
from .errors import DuplicateIDError, InvalidAgeError

type StudentRecord = tuple[int, str, str, int, str]


class StudentTable:
    def __init__(self) -> None:
        self._student: list[StudentRecord] = []

    def create_record(
        self,
        student_id: int,
        first_name: str,
        second_name: str,
        age: int,
        sex: str,
    ) -> StudentRecord:

        if age < 0:
            raise InvalidAgeError("Поле age не может быть отрицательным.")

        if any(record[0] == student_id for record in self._student):
            raise DuplicateIDError(f"Запись с id={student_id} уже существует.")

        new_record: StudentRecord = (
            student_id,
            first_name.strip(),
            second_name.strip(),
            age,
            sex.strip(),
        )
        self._student.append(new_record)
        return new_record
```

### Расширение набора тестов

Для уменьшения объёма дублируемого кода в тестах целесообразно использовать метод `setUp`. Дополнительно можно расширить набор тестовых данных и применять `subTest` для табличного тестирования.

```python
# tests/test_memory.py
import unittest
from src.db.backend.memory import StudentTable
from src.db.backend.errors import InvalidAgeError, DuplicateIDError


class TestMemory(unittest.TestCase):

    def setUp(self):
        self.student_table = StudentTable()
        self.assertIsInstance(self.student_table, StudentTable)

    def test_create_record(self):
        cases = [
            (1, "John", "Doe", 20, "M"),
            (2, "Jane", "Smith", 22, "F"),
            (3, "Alice", "Johnson", 19, "F"),
            (4, "Bob", "Brown", 21, "M"),
            (5, "Charlie", "Davis", 18, "M"),
            (6, "Eve", "Miller", 23, "F"),
            (7, "Frank", "Wilson", 20, "M"),
            (8, "Grace", "Moore", 22, "F"),
            (9, "Hank", "Taylor", 19, "M"),
            (10, "Ivy", "Anderson", 21, "F"),
            (11, "Jack", "Thomas", 18, "M"),
            (12, "Kathy", "Jackson", 23, "F"),
            ]

        for test_data in cases:
            # Используем subTest для изоляции каждого тестового случая и улучшения читаемости результатов тестирования.
            # Это позволяет нам видеть, какой именно набор данных вызвал ошибку, если тест не пройдет.
            with self.subTest(test_data=test_data):
                record = self.student_table.create_record(*test_data)
                self.assertEqual(record, test_data)

    def test_create_record_negative_age(self):
        cases = [
            (1, "John", "Doe", -1, "M"),
            (2, "Jane", "Smith", -5, "F"),
            (3, "Alice", "Johnson", -10, "F"),
        ]
        error_message = "Поле age не может быть отрицательным."

        for test_data in cases:
            with self.subTest(test_data=test_data):
                with self.assertRaises(InvalidAgeError) as context:
                    self.student_table.create_record(*test_data)

        self.assertEqual(str(context.exception), error_message)

    def test_create_record_duplicate_id(self):
        test_data_1 = (1, "John", "Doe", 20, "M")
        test_data_2 = (1, "Jane", "Smith", 22, "F")
        error_message = "Запись с id=1 уже существует."

        self.student_table.create_record(*test_data_1)

        with self.assertRaises(DuplicateIDError) as context:
            self.student_table.create_record(*test_data_2)

        self.assertEqual(str(context.exception), error_message)    
```

Следует отметить, что данный подход делает тесты более компактными и удобными для сопровождения, особенно при увеличении количества проверяемых сценариев.

По аналогии с предыдущими этапами необходимо сформулировать тесты для метода `select_record`.

```python

import unittest
from src.db.backend.memory import StudentTable
from src.db.backend.errors import InvalidAgeError, DuplicateIDError


class TestMemory(unittest.TestCase):
    def setUp(self):

            (2, "Jane", "Smith", 22, "F"),
```

Данный набор тестов проверяет как полную выборку без фильтров, так и выборку по каждому отдельному критерию.

### Итоговая реализация модуля `memory.py`

После разработки и прохождения тестов итоговая реализация базы данных принимает следующий вид:

```python
from .errors import DuplicateIDError, InvalidAgeError

type StudentRecord = tuple[int, str, str, int, str]


class StudentTable:
    def __init__(self) -> None:
        self._student: list[StudentRecord] = []

    def create_record(
        self,
        student_id: int,
        first_name: str,
        second_name: str,
        age: int,
        sex: str,
    ) -> StudentRecord:

        if age < 0:
            raise InvalidAgeError("Поле age не может быть отрицательным.")

        if any(record[0] == student_id for record in self._student):
            raise DuplicateIDError(f"Запись с id={student_id} уже существует.")

        new_record: StudentRecord = (
            student_id,
            first_name.strip(),
            second_name.strip(),
            age,
            sex.strip(),
        )
        self._student.append(new_record)
        return new_record

    def select_record(
        self,
        student_id: int | None = None,
        first_name: str | None = None,
        second_name: str | None = None,
        age: int | None = None,
        sex: str | None = None,
    ) -> list[StudentRecord]:

        if (
            student_id is None
            and first_name is None
            and second_name is None
            and age is None
            and sex is None
        ):
            return self._student.copy()

        result: list[StudentRecord] = []

        for record in self._student:
            if student_id is not None and record[0] != student_id:
                continue

            if first_name is not None and record[1] != first_name:
                continue

            if second_name is not None and record[2] != second_name:
                continue

            if age is not None and record[3] != age:
                continue

            if sex is not None and record[4] != sex:
                continue

            result.append(record)

        return result
```
