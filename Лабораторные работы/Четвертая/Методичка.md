## Введение

В третьей лабораторной работе система управления данными была переведена на объектно-ориентированную архитектуру. Это было сделано не только ради аккуратного кода, но и для того, чтобы в дальнейшем можно было заменить способ хранения данных без полного переписывания программы. Именно этот шаг и выполняется в текущей лабораторной работе.

Цель данной методички состоит в том, чтобы показать базовый вариант решения: добавить в проект файловую реализацию СУБД, сохранив единый интерфейс работы с данными. В качестве примера далее используется текстовый формат `JSON`, поскольку он прост для чтения, поддерживается стандартной библиотекой Python и хорошо подходит для учебной задачи.

Информация по возможной архитектуре проекта была изоложена в [шестой лекции](/Лекции/Шестая.md).

## Основная часть

### Что требуется сделать в этой лабораторной работе

На базовом уровне необходимо расширить проект из предыдущей лабораторной работы так, чтобы программа умела работать не только с `in-memory` хранилищем, но и с файловой СУБД.

Иными словами, после выполнения работы система должна:

1. создавать таблицу;
2. добавлять записи;
3. читать записи с фильтрацией;
4. сохранять данные на диск;
5. загружать данные из файлов при следующем запуске программы.

### Теоретические основы работы с файлами

При работе с программами данные могут храниться либо только в оперативной памяти, либо во внешнем хранилище. В предыдущих лабораторных работах использовалась `in-memory` модель: пока программа запущена, данные существуют, но после завершения работы процесса они теряются. Файлы решают именно эту проблему, поскольку позволяют сохранять состояние программы на диске.

С практической точки зрения файл можно понимать как именованную область внешней памяти, в которую программа может записывать данные и из которой она может затем их читать. Для нашей лабораторной работы это означает, что таблица базы данных больше не обязана жить только внутри объекта Python. Её состояние можно выгрузить в файл, а затем восстановить при следующем запуске.

### Открытие файла

В Python работа с файлами обычно начинается с функции `open`. Она принимает путь к файлу и режим работы.

Наиболее употребимые режимы:

- `r` - чтение файла;
- `w` - запись файла с перезаписью содержимого;
- `a` - добавление данных в конец файла;
- `x` - создание нового файла с ошибкой, если он уже существует.

Также часто используется параметр `encoding="utf-8"`, особенно если в данных присутствует кириллица.

Пример чтения текстового файла:

```python
with open("data.txt", "r", encoding="utf-8") as file:
    content = file.read()
```

Пример записи:

```python
with open("data.txt", "w", encoding="utf-8") as file:
    file.write("Пример текста")
```

### Контекстный менеджер `with`

Файлы важно не только открыть, но и корректно закрыть после завершения работы. Если этого не делать, можно получить утечки ресурсов или некорректную запись данных.

Именно поэтому в Python обычно используется конструкция `with`. Она автоматически закрывает файл после завершения блока, даже если внутри возникло исключение.

Это особенно важно для нашей лабораторной работы, так как файловая СУБД регулярно выполняет операции чтения и записи. Чем меньше таких технических деталей остаётся "на совести" программиста, тем надёжнее получается программа.

### Текстовые и бинарные файлы

Файлы можно условно разделить на текстовые и бинарные.

Текстовые файлы хранят данные в виде символов, которые можно открыть обычным редактором. К таким форматам относятся `txt`, `csv`, `json`, `xml`. Бинарные файлы хранят данные в специальном представлении, которое обычно неудобно читать человеку напрямую.

Для учебной файловой СУБД разумнее использовать именно текстовый формат, потому что:

1. его легко проверять вручную;
2. его проще отлаживать;
3. он лучше подходит для демонстрации структуры данных.

### Почему для лабораторной работы удобен `JSON`

Формат `JSON` хорошо подходит для хранения таблиц, потому что он естественно отображает словари, списки, строки, числа и логические значения. А именно из таких сущностей и состоит наша учебная база данных.

Например, запись студента в Python:

```python
{
    "student_id": 1,
    "first_name": "Иван",
    "age": 20
}
```

почти без изменений переносится в `JSON`.

В стандартной библиотеке Python для этого есть модуль `json`. Основные функции:

- `json.dump(...)` - записывает Python-объект в файл;
- `json.load(...)` - читает объект из файла;
- `json.dumps(...)` - преобразует объект в строку;
- `json.loads(...)` - создаёт объект из строки.

Для нашей задачи чаще всего используются именно `json.dump` и `json.load`, поскольку мы работаем не просто со строками, а именно с файлами.

### Что именно нужно сохранять в файловой СУБД

Чтобы восстановить таблицу после перезапуска программы, недостаточно сохранить только список записей. Нужно сохранить и схему таблицы, то есть набор колонок.

Поэтому в файле целесообразно хранить как минимум две части:

1. `columns` - список имён столбцов;
2. `records` - список записей.

Это важно с архитектурной точки зрения. Таблица - это не просто набор строк, а структура данных с определёнными правилами. Если не сохранить схему, то при повторной загрузке программа уже не сможет корректно проверить, соответствует ли новая запись структуре таблицы.

### Возможные ошибки при работе с файлами

Работа с файлами почти всегда связана с дополнительными рисками по сравнению с `in-memory` хранением. Например:

1. файл может отсутствовать;
2. файл может содержать повреждённые или некорректные данные;
3. каталог для хранения данных может ещё не существовать;
4. пользователь может попытаться загрузить таблицу, которая не была создана.

Именно поэтому файловая СУБД должна содержать проверки и понятные сообщения об ошибках. По аналогии с третьей лабораторной работой такие ситуации лучше оформлять через пользовательские исключения, а не через общий `ValueError`. Это делает код понятнее и упрощает тестирование.

### Идея решения

Ключевая идея берётся из шестой лекции: пользовательский код должен работать с единым интерфейсом базы данных, а конкретная реализация может быть разной.

Это означает, что удобно выделить:

1. класс `Table`, который отвечает за структуру таблицы и операции над её записями;
2. абстрактный класс `Database`, который задаёт общий интерфейс;
3. класс `MemoryDatabase`, который хранит данные в памяти;
4. класс `FileDatabase`, который сохраняет таблицы в файлы.

Тогда пользовательский интерфейс будет вызывать одни и те же методы независимо от того, где физически лежат данные.

### Выбор формата хранения

Для базового решения удобно хранить каждую таблицу в отдельном `JSON`-файле. В таком файле можно сохранить:

- список колонок;
- список записей.

Например, таблица `students` может храниться в файле `data/students.json`:

```json
{
  "columns": ["student_id", "first_name", "second_name", "age", "sex"],
  "records": [
    {
      "student_id": 1,
      "first_name": "Иван",
      "second_name": "Иванов",
      "age": 20,
      "sex": "М"
    }
  ]
}
```

Такой подход хорош тем, что:

1. данные читаются человеком;
2. структура файла соответствует структуре таблицы;
3. для работы достаточно стандартного модуля `json`.

### Пользовательские исключения

Как и в третьей лабораторной работе, в этой задаче удобно вынести типовые ошибки в отдельный модуль `errors.py`. Это позволяет не смешивать бизнес-логику с обработкой исключительных ситуаций и делает тесты более выразительными.

Для базового решения можно использовать следующий набор ошибок:

```python
# src/db/backend/errors.py
class DatabaseError(Exception):
    """Базовый класс для ошибок базы данных."""


class TableAlreadyExistsError(DatabaseError):
    """Ошибка, возникающая при попытке создать уже существующую таблицу."""


class TableNotFoundError(DatabaseError):
    """Ошибка, возникающая при обращении к несуществующей таблице."""


class MissingColumnError(DatabaseError):
    """Ошибка, возникающая при отсутствии обязательного поля в записи."""


class UnknownColumnError(DatabaseError):
    """Ошибка, возникающая при использовании поля, которого нет в схеме таблицы."""


class InvalidStorageDataError(DatabaseError):
    """Ошибка, возникающая при чтении повреждённых данных из файла."""
```

Такой подход полезен по двум причинам:

1. сразу видно, какая именно ситуация произошла;
2. в тестах можно проверять конкретный тип ошибки, а не только текст сообщения.

### Реализация класса `Table`

Начать удобнее всего с класса таблицы, поскольку и `MemoryDatabase`, и `FileDatabase` будут работать именно с ним.

```python
# src/db/backend/table.py
from typing import Any

from .errors import MissingColumnError, UnknownColumnError


class Table:
    """Таблица с фиксированным набором колонок."""

    def __init__(self, columns: tuple[str, ...], records: list[dict[str, Any]] | None = None) -> None:
        self.columns = columns
        self.records: list[dict[str, Any]] = []

        if records is not None:
            for record in records:
                self.insert_record(record)

    def insert_record(self, record: dict[str, Any]) -> None:
        """Добавляет запись, если она соответствует схеме таблицы."""
        missing_columns = [column for column in self.columns if column not in record]
        if missing_columns:
            raise MissingColumnError(
                f"Отсутствует поле '{missing_columns[0]}' в записи."
            )

        extra_columns = [column for column in record if column not in self.columns]
        if extra_columns:
            raise UnknownColumnError(
                f"Поле '{extra_columns[0]}' не определено в структуре таблицы."
            )

        self.records.append(record.copy())

    def select_records(self, **filters: Any) -> list[dict[str, Any]]:
        """Возвращает записи, удовлетворяющие всем переданным фильтрам."""
        unknown_filters = [key for key in filters if key not in self.columns]
        if unknown_filters:
            raise UnknownColumnError(
                f"Поле '{unknown_filters[0]}' не определено в структуре таблицы."
            )

        if not filters:
            return [record.copy() for record in self.records]

        result: list[dict[str, Any]] = []
        for record in self.records:
            if all(record.get(key) == value for key, value in filters.items()):
                result.append(record.copy())

        return result
```

В таком варианте класс `Table` отвечает только за структуру таблицы и операции над записями. Логику преобразования в `JSON` удобнее вынести в файловую реализацию базы данных, поскольку это уже относится не к модели таблицы, а к способу хранения данных.

### Реализация общего интерфейса `Database`

Далее удобно вынести общую логику работы с таблицами в абстрактный класс.

```python
# src/db/backend/database.py
from abc import ABC, abstractmethod
from typing import Any

from .errors import TableAlreadyExistsError
from .table import Table


class Database(ABC):
    """Общий интерфейс базы данных."""

    def create_table(self, table_name: str, columns: tuple[str, ...]) -> None:
        if self._table_exists(table_name):
            raise TableAlreadyExistsError(
                f"Таблица '{table_name}' уже существует."
            )

        self._save_table(table_name, Table(columns))

    def insert_record(self, table_name: str, record: dict[str, Any]) -> None:
        table = self._load_table(table_name)
        table.insert_record(record)
        self._save_table(table_name, table)

    def select_records(self, table_name: str, **filters: Any) -> list[dict[str, Any]]:
        table = self._load_table(table_name)
        return table.select_records(**filters)

    @abstractmethod
    def _table_exists(self, table_name: str) -> bool:
        """Проверяет наличие таблицы."""

    @abstractmethod
    def _load_table(self, table_name: str) -> Table:
        """Загружает таблицу."""

    @abstractmethod
    def _save_table(self, table_name: str, table: Table) -> None:
        """Сохраняет таблицу."""
```

Такой класс задаёт публичный интерфейс, а детали хранения скрываются внутри служебных методов `_table_exists`, `_load_table` и `_save_table`.

### Реализация `MemoryDatabase`

Для полноты архитектуры приведём и `in-memory` реализацию в новом стиле:

```python
# src/db/backend/memory.py
from .database import Database
from .errors import TableNotFoundError
from .table import Table


class MemoryDatabase(Database):
    """База данных, хранящая таблицы в оперативной памяти."""

    def __init__(self) -> None:
        self.tables: dict[str, Table] = {}

    def _table_exists(self, table_name: str) -> bool:
        return table_name in self.tables

    def _load_table(self, table_name: str) -> Table:
        if table_name not in self.tables:
            raise TableNotFoundError(
                f"Таблица '{table_name}' не существует."
            )

        return self.tables[table_name]

    def _save_table(self, table_name: str, table: Table) -> None:
        self.tables[table_name] = table
```

Эта реализация почти не отличается от той, что уже обсуждалась в шестой лекции.

### Реализация `FileDatabase`

Теперь можно перейти к основной части лабораторной работы.

```python
# src/db/backend/file.py
import json
from pathlib import Path

from .database import Database
from .errors import InvalidStorageDataError, TableNotFoundError
from .table import Table


class FileDatabase(Database):
    """База данных, которая хранит таблицы в JSON-файлах."""

    def __init__(self, directory: str = "data") -> None:
        self.directory = Path(directory)
        self.directory.mkdir(parents=True, exist_ok=True)

    def _table_exists(self, table_name: str) -> bool:
        return self._get_table_path(table_name).exists()

    def _load_table(self, table_name: str) -> Table:
        table_path = self._get_table_path(table_name)
        if not table_path.exists():
            raise TableNotFoundError(
                f"Таблица '{table_name}' не существует."
            )

        try:
            with table_path.open("r", encoding="utf-8") as file:
                data = json.load(file)
        except json.JSONDecodeError as error:
            raise InvalidStorageDataError(
                "Файл таблицы содержит некорректный JSON."
            ) from error

        return self._deserialize_table(data)

    def _save_table(self, table_name: str, table: Table) -> None:
        table_path = self._get_table_path(table_name)

        with table_path.open("w", encoding="utf-8") as file:
            json.dump(
                self._serialize_table(table),
                file,
                ensure_ascii=False,
                indent=2,
            )

    def _get_table_path(self, table_name: str) -> Path:
        return self.directory / f"{table_name}.json"

    def _serialize_table(self, table: Table) -> dict:
        return {
            "columns": list(table.columns),
            "records": [record.copy() for record in table.records],
        }

    def _deserialize_table(self, data: dict) -> Table:
        if "columns" not in data or "records" not in data:
            raise InvalidStorageDataError(
                "Файл таблицы имеет некорректную структуру."
            )

        columns = tuple(data["columns"])
        records = data.get("records", [])
        return Table(columns, records)
```

Разберём, что здесь происходит:

1. В конструкторе создаётся каталог для хранения таблиц.
2. Метод `_get_table_path` строит путь к файлу конкретной таблицы.
3. Метод `_table_exists` проверяет наличие файла.
4. Метод `_load_table` читает `JSON`, а при повреждённом файле возбуждает пользовательское исключение.
5. Метод `_save_table` преобразует таблицу в словарь и записывает его в файл.
6. Методы `_serialize_table` и `_deserialize_table` инкапсулируют всю файловую логику преобразования данных.

Обратите внимание, что и здесь используются пользовательские исключения. Это соответствует подходу из третьей лабораторной работы: ошибки являются частью архитектуры, а не случайными строками, разбросанными по коду.

Именно в этом и состоит базовая файловая СУБД: логика работы с таблицей остаётся общей, а меняется только способ хранения.

### Подключение файловой базы данных в пользовательском интерфейсе

Чтобы пользователь мог выбрать реализацию базы данных, интерфейс можно обновить следующим образом:

```python
# src/db/tui.py
from src.db.backend.file import FileDatabase
from src.db.backend.memory import MemoryDatabase


class TUI:
    def __init__(self) -> None:
        print("Выберите тип базы данных:")
        print("1. In-memory")
        print("2. File database")

        choice = input("Введите номер: ").strip()
        if choice == "2":
            self.database = FileDatabase()
        else:
            self.database = MemoryDatabase()
```

После этого остальная часть интерфейса может работать без изменений, если она обращается только к методам `create_table`, `insert_record` и `select_records`.

### Тестирование файловой реализации

Так как в третьей лабораторной работе уже были введены автоматизированные тесты, новую функциональность тоже следует покрыть тестами. Для файловой реализации удобно использовать временный каталог.

```python
# tests/test_file_database.py
import tempfile
import unittest

from src.db.backend.errors import TableNotFoundError
from src.db.backend.file import FileDatabase


class TestFileDatabase(unittest.TestCase):
    def test_data_is_saved_between_instances(self) -> None:
        with tempfile.TemporaryDirectory() as directory:
            first_db = FileDatabase(directory)
            first_db.create_table("students", ("student_id", "name"))
            first_db.insert_record(
                "students",
                {"student_id": 1, "name": "Иван"},
            )

            second_db = FileDatabase(directory)
            records = second_db.select_records("students")

            self.assertEqual(
                records,
                [{"student_id": 1, "name": "Иван"}],
            )

    def test_select_with_filters(self) -> None:
        with tempfile.TemporaryDirectory() as directory:
            db = FileDatabase(directory)
            db.create_table("students", ("student_id", "name"))
            db.insert_record("students", {"student_id": 1, "name": "Иван"})
            db.insert_record("students", {"student_id": 2, "name": "Мария"})

            records = db.select_records("students", name="Мария")

            self.assertEqual(
                records,
                [{"student_id": 2, "name": "Мария"}],
            )

    def test_select_from_missing_table(self) -> None:
        with tempfile.TemporaryDirectory() as directory:
            db = FileDatabase(directory)

            with self.assertRaises(TableNotFoundError):
                db.select_records("students")
```

Такой тест особенно важен, потому что он проверяет главное отличие файловой СУБД от `in-memory` реализации: данные должны сохраняться между двумя разными экземплярами программы.

## Заключение

В рамках данной лабораторной работы был рассмотрен базовый способ добавления файловой СУБД в уже существующий проект.
